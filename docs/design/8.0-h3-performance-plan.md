# HTTP/3 Fast-Path and Performance Plan

Reference: [2.0-architecture.md](2.0-architecture.md), [1.3-invariants.md](1.3-invariants.md), [7.0-rfc-compliance-index.md](7.0-rfc-compliance-index.md).

## Purpose

Track the engineering work required to take h3 from "correctness-in-progress on a smoke test" to "fastest h3 on HttpArena." This doc captures the audit findings, architectural decisions, and PR sequence that underlie that work so a future session can pick up without re-deriving the trace or the reasoning.

This is not a compliance document — RFC tracking lives in [7.0-rfc-compliance-index.md](7.0-rfc-compliance-index.md). This is a plan-of-record for the h3 hot path.

## Goal

Submit swerver to HttpArena's h3 profiles and top both the **Production** and **Tuned** tiers for `/baseline2` (throughput) and `/static/*` (multi-file serving) — the two h3 profiles the benchmark currently runs.

### Baseline (2026-04-09, ReleaseFast, AES-128-GCM, docker loopback, `h2load -c100 -n200000`)

- **h3 throughput:** ~16 500 req/s, 0 failures
- **h3 sequential latency** (1 conn, 1 req at a time): ~7 600 req/s, mean 128 μs, p99.5 1.16 ms
- **h1 throughput on the same box for comparison:** ~149 000 req/s — h3 is at ~11% of h1

The h3/h1 gap is unoptimized first-version cost: per-packet AEAD, HKDF, frame parsing, dedupe bitmap, single-threaded event loop, no batching, no pre-encoding. Every win in the performance ranking below is designed to close this gap and then overshoot it. Baseline recorded via `swerver-benchmarks/scenarios/http3/run.sh` — debian:trixie loadgen image with h2load built against ngtcp2/nghttp3.

### Target (Linux, per-worker, on HttpArena's hardware class)

| Profile | Target |
|---------|--------|
| `/baseline2` GET throughput (64 conns × 128 par) | 500K+ req/s per worker |
| `/baseline2` GET p99 latency (64 conns × 128 par) | < 2 ms |
| `/static/*` throughput (64 conns × 128 par, 360 KB dataset) | limited by NIC bandwidth, not CPU |

These aren't aspirational — they're what well-tuned h3 servers (msquic, quinn-with-batching, nginx-quic) already achieve on similar hardware. The target is to match or beat them. The 16 500 → 500K gap is ~30x; rank 1 (GSO) + rank 2 (recvmmsg) alone should deliver most of it.

## Non-goals

- Fast on macOS. macOS lacks UDP batching primitives (`recvmmsg`, `sendmmsg`, `UDP_SEGMENT`), so h3 will structurally lag Linux. Local dev only, benchmark on Linux.
- Kernel bypass (AF_XDP, DPDK). HttpArena's "Engine" tier allows it but is ranked separately and is not where swerver competes.
- Targeting h3 0-RTT resumption. Out of scope until Production/Tuned are landed.

## Architectural constraints (known, intentional)

- **Synchronous defer-until-FIN request handling.** The Stack buffers nothing until FIN; on FIN it parses everything in one pass and dispatches the full request to a synchronous router handler. This is the right call for a benchmark-oriented server against a synchronous router contract, but it **constrains future features**:
  - **Streaming uploads** (large file POSTs) buffer the entire body before dispatch. 413 on oversized bodies.
  - **WebSockets over HTTP/3** (RFC 9220) cannot fit this model — WebSockets need per-frame dispatch from an open request stream.
  - **Server-sent events on the request side** (handler reading from a live client stream) cannot fit.
  - Adding any of these means revisiting the defer-until-FIN design and building a streaming handler contract in the router.
- **Multi-DATA-frame request bodies are rejected in PR A** with `error.ConnectionError`. Single-DATA-frame bodies (the common case) work zero-copy. HttpArena h3 profiles are all GETs, so this limitation is invisible to the submission. Extending to multi-frame requires either in-place compaction (needs a mutable input buffer contract) or a bounded Stack-owned scratch slab (small copy cost).

## Load model

HttpArena's h3 profiles use `oha` as the load generator against a framework-provided Dockerfile. The load profile is:

- **Connections:** 64 OR 512 (two separate scenarios)
- **Parallelism:** 128 concurrent requests per connection (HTTP/3 stream pipelining)
- **Duration:** 5 seconds per run, 3 runs, best result taken
- **Scoring:** throughput + latency
- **Worst case concurrency per worker:** 512 conns × 128 streams = **65 536 concurrent h3 streams**

This concurrency number is load-bearing for the whole design — per-stream state must be tiny or lazy.

## Invariants

Two invariants the hot path must respect. Violations are blockers, not performance issues.

1. **Zero per-request heap allocations in steady state.** The steady-state request path must not call `allocator.alloc` / `AutoHashMap.put` / `ArrayList.appendSlice` (when growing past capacity). Fixed-size structures, buffer pool slabs, and `clearRetainingCapacity` patterns are allowed. Inherited from [1.3-invariants.md](1.3-invariants.md) and `docs/design/7.0-rfc-compliance-index.md` line 11.
2. **Zero body copies on the fast path.** From AEAD decrypt to handler dispatch, body bytes should be read once (as a slice into the decrypted packet buffer) and never moved. Headers may be transformed via QPACK decoding (unavoidable), but DATA frame bytes flow by reference.

A regression test enforcing invariant 1 lands with PR A: drive 1000 requests through a counting allocator, assert bounded allocation count.

## Audit findings — h3 request-stream buffering and body dispatch

These are the seven findings from the line-level audit performed in session
`/Users/justin/.claude/projects/-Users-justin-swerver/` against `main@33a6b0d`.
Line references are valid at that commit; update if the files move.

1. **Dead `StreamState` hashmap.** `src/protocol/http3.zig:121-132` defines `StreamState` and `src/protocol/http3.zig:194` declares `streams: AutoHashMap(u64, StreamState)`, init'd at line 235, deinit'd at 267-272, **never populated anywhere**. The `buffer` and `body_buffer` fields were the intended home for cross-`ingest` frame buffering and per-stream body accumulation. Wire was never completed. Delete or resurrect — either way, the current state is misleading to readers.

2. **`need_more` is discarded.** `Stack.processRequestStream` at `src/protocol/http3.zig:400-473` returns `{consumed, need_more: true}` on `error.UnexpectedEnd` but saves nothing. `Connection.processHttp3Stream` at `src/quic/connection.zig:640-646` returns only `result.events`, dropping `consumed` and `need_more`. `Handler.processStreamFrame` at `src/quic/handler.zig:563` fully discards the return value. **Consequence:** h3 frames split across QUIC STREAM frames corrupt the parser — the tail is never re-fed.

3. **`stream_frame.data` vs reassembled stream.** `Handler.processStreamFrame` at `src/quic/handler.zig:513-566` calls `s.receive(stream_frame.offset, stream_frame.data, stream_frame.fin)` (correct) then `conn.processHttp3Stream(stream_frame.stream_id, stream_frame.data, fin_now)` — feeding the RAW STREAM-frame payload, not the in-order reassembled bytes. The handler.zig:542-544 "recv_offset didn't advance" guard is a band-aid for pure-duplicate frames but does not handle partial overlaps or out-of-order → in-order catch-up.

4. **Double body accumulation.** `Stack.processRequestStream` at `src/protocol/http3.zig:446` calls `self.accumulateBody(stream_id, d.data)` on every DATA frame. The resulting `DataEvent` bubbles up to `Server.handleDatagram` at `src/server.zig:530-538`, which calls `stack.accumulateBody(data_ev.stream_id, data_ev.data)` AGAIN on the same bytes. Every DATA byte is appended twice. Latent because `handleHttp3Request` is always called with `body = ""` today, so the accumulated garbage is discarded before anyone reads it — but any future body-dispatch patch will deliver 2× bodies until this is fixed.

5. **Body-bearing requests silently dropped.** `src/server.zig:544-553` dispatches `handleHttp3Request` only when `hdrs.end_stream == true`. For POST/PUT with a body, end_stream is false on HEADERS, so dispatch never happens. The subsequent DATA + FIN events at `src/server.zig:554-567` only call `clearRequestBody` — the request is silently discarded. Comment at lines 549-552 admits this: "POST/PUT over HTTP/3 with multi-packet bodies will get a 501 from the router" (actually no 501 — the client times out). There is no per-stream `HeadersEvent` persistence anywhere in the Stack, so even if someone fixed the dispatch trigger, the headers would be long gone by the time the body completes.

6. **Duplicate EndStreamEvent.** On a FIN packet whose last h3 frame is DATA, `Stack.processRequestStream` emits `DataEvent{end_stream=true}` (lines 448-454) AND an `EndStreamEvent` (lines 462-466). The server processes both in the second loop: `data_ev.end_stream` → `clearRequestBody` (`src/server.zig:558`), then `.end_stream` → `clearRequestBody` again (`src/server.zig:565`). Two events, overlapping semantics, neither authoritative.

7. **`copyHeaders` clobbers on the next call.** `Stack.header_buf` / `header_name_storage` / `header_value_storage` at `src/protocol/http3.zig:198-204` are fixed-size instance fields. `copyHeaders` at lines 242-264 resets `self.header_count = 0` on every call and overwrites from the start. If two streams emit HEADERS in the same `ingest` call (plausible for 128-pipelined requests), the second clobbers the first. Today this is masked because `handleHttp3Request` is called synchronously and finishes before the next `ingest`, but any future code holding a `HeadersEvent.headers` slice past the next `ingest` call reads corrupted memory.

## Adjacent findings (not in the above audit, but uncovered while doing it)

- **h2 POST/PUT bodies are also discarded.** `src/server.zig:1177-1179` has `.data => |data| { _ = data; },`. `hdr.request` at `src/server.zig:1162` has body="" (h2's `HeadersEvent` carries a `RequestView` with empty body). Same silent-drop bug as h3, on a different protocol. Deserves its own known-issue row in the compliance index under RFC 9113. Part of the memory-noted benchmark gap (swerver 12K vs actix 33K on Mixed GET+POST) is likely this.

- **`Connection.processPacket` is dead code.** `src/quic/connection.zig:976` and `processShortHeaderPacket` at line 1026 have zero callers. Grep `.processPacket(` — only the `Handler` variant at `src/quic/handler.zig:89` is used in production. Delete in PR C.

- **Crypto wrappers double-copy every packet.** `unprotectPacket` at `src/quic/crypto.zig:495-547` decrypts into a 64 KiB stack buffer `plaintext_buf`, then memcpys back into `packet_bytes[header_len..]`. `protectPacket` at lines 457-490 does the symmetric thing with `ciphertext_buf`. The inner `unprotectPayload`/`protectPayload` at lines 381 and 417 already take an `out: []u8` parameter — the waste is entirely in the outer wrapper. Fix is ~3 lines: pass `packet_bytes[header_len..]` directly to the inner function. Savings: 1 memcpy per encrypt + 1 per decrypt (~1500 B each) + elimination of 64 KiB stack thrash per call. Tracked as PR PERF-4.

- **`Stream.receive` out-of-order path allocates.** `src/quic/stream.zig:200` calls `allocator.alloc(data.len)` + memcpy to queue `pending_chunks` for out-of-order STREAM frames. Heap alloc on the (rare) reorder path. Not steady-state hot but is an invariant violation in the general case. Fix: pool-backed `pending_chunks`.

- **`Stream.recv_buffer` starts at zero capacity.** `src/quic/stream.zig:75` initializes with `.empty`. First `appendSlice` heap-allocs. Needs `ensureTotalCapacity` at `Stream.init` — but the size has to be careful because with 65 K concurrent streams, big per-stream pre-allocations are infeasible. Lazy slab-backed storage from the buffer pool is the right answer.

## Architectural decisions

### Defer-until-FIN parsing with body as a slice

h3 request-stream handling runs **only when FIN is seen**. Until then, the Stack doesn't touch the request. On FIN, a single `ingest` call parses all frames in one pass, QPACK-decodes headers, identifies the DATA frame payload slice, and emits one event.

This collapses `HeadersEvent + DataEvent + EndStreamEvent` into a single `RequestReadyEvent { stream_id, headers, body }`. The router handler signature at `src/router/router.zig:26` (`HandlerFn = *const fn (ctx: *HandlerContext) response.Response`) is synchronous and consumes full requests — there is no streaming API, no reason to fire per-frame events.

Rationale: eliminates the per-HeadersEvent state persistence problem (finding #5 and #7), eliminates the duplicate end-stream signal (finding #6), makes the allocation budget trivial to reason about.

### Inline fast path for single-STREAM-frame GETs

`/baseline2` — the entire HttpArena h3 throughput benchmark — is a trivial GET whose HEADERS frame fits in one QUIC STREAM frame with FIN set. For this case, `Handler.processStreamFrame` bypasses `Stream.recv_buffer` entirely:

```
if (stream_is_new and stream_frame.offset == 0 and stream_frame.fin
    and stream_is_bidi_client_request)
{
    // Parse h3 frames in-place on stream_frame.data (which points into
    // the decrypted UDP packet buffer).
    // QPACK-decode into per-worker scratch.
    // Dispatch to router synchronously.
    // Encode response into per-worker send scratch.
    // Let the caller's GSO batcher flush.
    return;
}
// Slow path: s.receive → s.read → stack.ingest → consumeRead
```

Zero per-stream allocation, zero per-request allocation, no `recv_buffer`, no `pending_slots`. This is the path `/baseline2` always takes.

### Slow path uses `Stream.read()` + `Stream.consumeRead()`

When the fast path doesn't apply (multi-frame HEADERS, body-bearing request, any fragmented frame), the handler calls `s.receive`, then `const contiguous = s.read()`, feeds it to `conn.processHttp3Stream`, and calls `s.consumeRead(result.consumed)` to drop parsed bytes from the Stream's buffer. Unparsed tail persists in `recv_buffer` across calls.

This fixes findings #2 and #3 in one move and makes the `StreamState.buffer` field obsolete (finding #1 resolved for free). `s.consumeRead` at `src/quic/stream.zig:249` exists and has zero production callers today — greenfield contract.

### `pending_slots` instead of `AutoHashMap`

For slow-path requests that need per-stream state across `ingest` calls, use a fixed-size `pending_slots: [MAX_PENDING_H3_REQUESTS]PendingSlot` array per Stack, where `MAX_PENDING_H3_REQUESTS = 128` matches the h2 max_streams default. Lookup is linear scan over a cache-hot 128-entry array. Zero allocations, zero hashmap overhead. Eviction on dispatch or stream reset.

This mirrors `BodyAccumState` in `src/runtime/connection.zig:12-26` — the h1 path already does this pattern.

### Stream.recv_buffer is lazy, slab-backed

For the slow path, `Stream.recv_buffer` acquires a buffer pool slab on first write and releases on last consume. Most streams — hit by the fast path — never touch it at all. At 65 K concurrent streams this means ~0-N slabs live at any given time, not 65 K × pre-sized-window.

## RFC correctness prerequisites (must be addressed before production)

The following are **known issues** from [7.0-rfc-compliance-index.md](7.0-rfc-compliance-index.md) that don't show on localhost/docker benchmarks but will sink any real-network deployment. They're not on the performance PR list because they don't block the HttpArena submission, but they MUST be fixed before swerver runs anywhere with a real NIC and real packet loss. Review feedback flagged this as the #1 risk; captured here to make sure it's tracked.

- **RFC 9002 loss detection / retransmission: Recovery.onPacketSent has zero production callers.** The live send path never records sent packets, so timer-based and packet-threshold loss detection never fire. On loopback this is invisible; on any path with > 0% packet loss the connection wedges and the peer sees silence. **Status: must fix before production.**
- **RFC 9000 ACK range parsing: processAckFrame computes ranges, Recovery.onAckReceived cumulatively marks `[0, largest_acked]`.** ACKs with gaps (normal under light loss) incorrectly retire unacked packets. Locks up the send side. **Status: must fix before production.**
- **RFC 9000 MAX_DATA / MAX_STREAM_DATA / PATH_RESPONSE not emitted.** Long-lived connections stall when the peer's send window exhausts. Short request/response patterns on HttpArena don't exercise this, but any sustained upload will. **Status: must fix before production.**
- **RFC 9000 draining not honored.** Handler removes non-alive connections from the pool immediately instead of keeping them for the draining period. Peer sees ICMP port-unreachable or similar. **Status: must fix before production; minor compared to loss detection.**
- **RFC 9001 key discard.** `onHandshakeComplete` clears `initial_space.keys` / `handshake_space.keys` but send/receive paths use `crypto_ctx.*` which still has live keys. Technically we can still decrypt Initial packets after the handshake is done. **Status: minor hardening, fix before production.**
- **RFC 9204 QPACK silent processing of dynamic table inserts.** We advertise `qpack_max_table_capacity = 0` in SETTINGS but `Stack.init` still creates a 4096-byte decoder table and `processQpackEncoderStream` accepts peer Insert instructions without rejection. A non-conformant peer can bypass our advertised limit. **Status: small fix, should land soon — reject Insert if capacity == 0.**

These are tracked in [7.0-rfc-compliance-index.md](7.0-rfc-compliance-index.md) but were not reflected in earlier drafts of this document's PR list. Treat the performance PR sequence below as incomplete without at least the first three of these landing before any non-benchmark traffic reaches the server.

**Sequencing notes (updated after review):**

- **PR PERF-4 moved up, PR INFRA promoted to parallel.** PR PERF-4 is independent and trivially saves 1 memcpy + 64 KiB stack thrash per packet. PR INFRA is the bottleneck of the measurement loop — land it the moment PR A is in, submit an initial non-competitive version, and iterate against real HttpArena numbers rather than docker-loopback. Loopback benchmarks mislead by an unknown factor.
- **PR PERF-0 is new.** `server.zig::buildStreamPacket` and `handler.zig::buildResponseFlight` are two parallel send paths with overlapping logic. PR PERF-1 (GSO) wants one unified send path to batch through — unifying them first avoids re-implementing GSO twice.
- **PR A is done and not split retroactively.** The review correctly noted PR A should ideally have been split into four (body dispatch on FIN / RequestReadyEvent consolidation / inline fast path / counting-allocator test). Taking the hit for PR A. Future correctness fixes WILL split more aggressively — call it lesson learned.
- **All PR PERF-* PRs require a Linux test loop.** GSO, recvmmsg, sendmmsg are no-ops on macOS. Do not try to land them without a Linux development environment.

Ordered by expected throughput impact on the httparena benchmarks:

## HttpArena submission facts

Critical recon that is not in the codebase and cost four WebFetch calls to discover. If the MDA2AV/HttpArena repo or the http-arena.com docs restructure, some of this may move — re-verify before the submission PR.

### H3 Baseline: `GET /baseline2?a=1&b=1`
- Port 8443, TLS + QUIC
- Response: `200`, `Content-Type: text/plain`, body `2` (literal sum)
- Shared endpoint with h2 `/baseline2`
- Load: 64 or 512 QUIC connections, 128 parallel requests per connection, 5s × 3 runs
- Load generator: `oha`
- Validation: reuses h2 `/baseline2` validator (no h3-specific checks)

### H3 Static Files: `GET /static/*`
- Port 8443, TLS + QUIC
- 20 files, ~360 KB total, mixed types (CSS, JS, HTML, fonts, SVG, WebP, JSON)
- URIs cycle through `requests/static-h2-uris.txt`
- Correct `Content-Type` per file extension required
- Load: same 64/512 conns × 128 par, 5s × 3 runs, oha

### Submission tiers
- **Production:** "native QUIC/HTTP3 support with default configuration." Static files **must** load from disk on every request — no mmap, no caching, no pre-loaded buffers. Batching primitives (GSO, recvmmsg) are allowed (not "caching").
- **Tuned:** QUIC params, congestion control, UDP buffer sizes, caching, mmap, pre-rendered headers, pre-encoded response packets all allowed.
- **Engine:** no constraints, kernel bypass allowed, ranked separately — not where swerver competes.

swerver submits to Production and Tuned. Build modes probably correspond to `-Doptimize=ReleaseFast` with / without a `-Dpreencode=true` flag gating the pre-encoded-response code path.

### Benchmark environment
- Docker `--network host`
- Threads: 64 (gcannon) / 128 (h2load) — oha's threading is separate
- Duration: 5s
- Runs: 3, best taken
- Data mounts: `/data/dataset.json`, `/data/benchmark.db`, `/certs/server.crt`

## Open questions (answer before / during implementation)

- **`initial_max_stream_data_bidi_remote` default.** Currently not verified. Needs to be small enough that 65 K concurrent streams don't blow memory, large enough that the h3 static-file responses can flow without stall. A lazy slab-backed `recv_buffer` sidesteps the sizing issue but makes `ensureTotalCapacity(initial_max_recv)` unnecessary.
- **`initial_max_streams_bidi` is currently hard-coded to 1 000 000** in `populateDefaultLocalParams`. Works for the 65 K streams benchmark case but wastes cycles advertising a limit we can't actually honor. Proper fix is per-connection stream-credit accounting + `MAX_STREAMS` frame emission. Not on the PR A critical path but file it.
- **Zig stdlib `Aes128Gcm.decrypt` in-place safety.** PERF-4 assumes `dst.ptr == src.ptr` works. Verify with a unit test before relying on it; if broken, fall back to a per-worker 2 KiB scratch.
- **`oha` h3 client behavior.** Check which features it exercises (0-RTT? key update? packet coalescing patterns?) so we know the minimum compatibility bar for the submission.
- **Static file dirfd caching across forks.** `c76b0a7` cached `static_root` as a dirfd. Verify that the fork model preserves the dirfd correctly for h3 workers (should — dirfds survive fork — but worth confirming with a test).
- **Memory footprint at 65 K streams × 512 conns.** Sketch the worst case before PR PERF-1 and budget the buffer pool accordingly. Current default `buffer_size=64KiB × buffer_count=4096 = 256 MiB` may need adjustment.
- **Debian image for the HttpArena Dockerfile.** Must be `debian:trixie-slim` or newer — OpenSSL 3.0.x on `bookworm` lacks `SSL_set_quic_tls_cbs`. Non-h3 builds were broken by unconditional FFI linkage until `f41d033` gated the wrapper bodies on `comptime build_options.enable_http3`. Do not drop that gate in the INFRA PR.
