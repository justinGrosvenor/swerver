# Core Runtime: Worker Scheduling and Affinity

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference architecture: [2.0-architecture.md](2.0-architecture.md).

## Scope

Defines the worker model, connection ownership, and scheduling rules for maximizing throughput and minimizing tail latency.

## Worker Model

- One event loop per core (or per performance core on hybrid systems).
- Each worker owns a disjoint connection pool subset.
- Connections are pinned to a worker for their lifetime.

## Accept Strategy

- Use `SO_REUSEPORT` to distribute accepts across workers. The listener setup (`net.zig setReusePort`) unconditionally attempts it, gated by `@hasDecl(std.posix.SO, "REUSEPORT")` on the platform; where the option is unsupported it is simply skipped (and the setsockopt failure is swallowed). There is **no** dedicated-acceptor / round-robin fallback.
- Multi-listener: each forked worker binds **all** configured listeners (`server.cfg.listeners[]`) via `SO_REUSEPORT`. With `workers: 0` the worker count defaults to `nproc`.

## Scheduling Rules

- The poll loop should block with a bounded timeout; avoid busy-wait.
- Prioritize write completions before scheduling new reads to reduce queue pressure.
- Limit per-connection work per poll to avoid starvation; use a fixed budget.

## Cross-Thread Handoff

- Avoid handoff; if necessary, use a lock-free ring buffer and transfer ownership explicitly.
- Handoff requires transferring connection state, buffers, and protocol context atomically.

## NUMA Considerations

- Pin workers to NUMA nodes where possible.
- Allocate per-worker pools to avoid cross-node memory traffic.

## Validation

- Benchmark with 1, N, and 2N workers to identify contention.
- Measure tail latency under write-heavy loads.
