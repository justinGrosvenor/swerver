# Core Runtime: Allocator and Memory Layout

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference invariants: [1.3-invariants.md](1.3-invariants.md).

## Scope

Defines the allocator strategy, memory layout, and sizing rules needed to ensure zero per-request heap allocation and predictable memory bounds.

## Allocator Strategy

Principles:

- Allocate all long-lived runtime structures at startup.
- Avoid heap allocation on the steady-state request path.
- Prefer contiguous allocations for cache locality.

Recommended model:

- Global allocator used only at startup.
- Fixed pools for connections, buffers, and per-connection queues.
- Optional per-thread arenas for protocol-local scratch, reset between polls.

## Memory Layout

```
Runtime Memory
  - ConnectionPool (max_connections)
  - BufferPool (buffer_count * buffer_size)
  - Per-connection queues (write queue, stream map)
  - Metrics/telemetry buffers
```

Cache guidance:

- Keep connection metadata in a compact struct (< 256 bytes).
- Store hot fields (state, counters, timestamps) in the first cache line.
- Store cold fields (peer address, protocol details) in a secondary struct or trailing fields.

## Sizing Formulas

```
connection_bytes = max_connections * sizeof(Connection)
buffer_bytes = buffer_count * buffer_size
write_queue_bytes = max_connections * write_queue_depth * sizeof(WriteEntry)
total_bytes = connection_bytes + buffer_bytes + write_queue_bytes + overhead
```

Guidance:

- `buffer_count >= max_connections * 2` to allow one read and one write buffer per connection.
- `buffer_size` should be tuned to typical header + small body sizes (16KB default).
- `write_queue_depth` should cover expected in-flight responses (4-16 typical).

## Ownership Rules

- Pools are owned by the runtime and freed only at shutdown.
- Protocol stacks may reference slices but must not retain them across poll boundaries.
- Any temporary allocations must come from a per-thread arena that is reset per poll.

## Safety Rules

- Bounds-check all buffer access in debug builds.
- Track outstanding buffer handles to detect leaks.
- Guard against ID overflow (u64 wrap) by refusing new connections and alerting.

## Validation

- Memory accounting test that sums pool sizes and matches configured bounds.
- Stress test under max connections to confirm no heap allocation in steady-state.
