# Core Runtime: File I/O and Zero-Copy Paths

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference architecture: [2.0-architecture.md](2.0-architecture.md).

## Scope

Defines how the runtime performs file-backed responses with zero-copy or minimal-copy paths across platforms.

## File Response Types

- Static file: full file or range response.
- Streamed file: incremental reads for very large files.

## Platform Strategies

Linux:

- Prefer `sendfile` for file-backed responses.
- Use `io_uring` with fixed buffers for async reads + socket writes when enabled.
- Use `pread` into pooled buffers as fallback.

macOS/BSD:

- Prefer `sendfile` where supported.
- Fallback to `pread` into pooled buffers.

Windows:

- Prefer `TransmitFile` or `WSASend` with file handles.
- Fallback to `ReadFile` into pooled buffers.

## Buffer Integration

- Zero-copy paths must bypass buffer pools when the OS supports direct send.
- When fallback is required, buffers come from the shared pool and are returned immediately after write completion.

## Precompressed Sibling Serving

- When a `.br` or `.gz` sibling of the requested file exists and the client's `Accept-Encoding` permits it (token present and not `q=0`), swerver serves the precompressed file directly with the matching `Content-Encoding` and `Vary: Accept-Encoding`.
- This is always on (no flag); it falls back to the identity file when no acceptable sibling exists. `resolveStaticVariant` tries `<path>.br` first, then `<path>.gz`.
- `Content-Type` is derived from the **original** request path, never the `.br`/`.gz` suffix.

## In-Memory Static Cache

- Optional, enabled by the `cache_static_files` config flag. Caches static file bodies in-process (per worker), keyed by path **and** encoding (identity / br / gzip).
- A cache hit skips the `open` / `fstat` / `sendfile` syscalls and serves the body straight from memory.
- The `Date` header stays fresh: it is supplied by the normal response path at send time, not baked into the cached bytes. Content-Length and HEAD handling likewise come from the normal path.

## Preencoded Error Responses

- A separate `preencoded` response cache stores fully-encoded canned responses (notably error responses) to avoid re-encoding on the hot path.
- Controlled by the positive `preencoded` config key (default on); internally this maps to a `disable_preencoded` flag. Error responses are cached by default.

## Range Requests

- Support single-range requests; multi-range may be deferred.
- Validate range bounds before scheduling I/O.
- Ensure range responses respect `max_body_bytes`.

## Caching and Metadata

- Cache file metadata (size, mtime) for hot paths.
- Use `etag`/`last-modified` to enable conditional requests.

## Backpressure

- File I/O counts toward write buffering limits.
- If write backpressure triggers, pause file reads and resume after drain.

## Validation

- Integration tests for small, large, and range responses.
- Performance tests comparing zero-copy vs fallback paths.
