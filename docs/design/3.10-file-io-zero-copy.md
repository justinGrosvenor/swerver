# Core Runtime: File I/O and Zero-Copy Paths

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference architecture: [2.0-architecture.md](2.0-architecture.md).

## Scope

Defines how the runtime performs file-backed responses with zero-copy or minimal-copy paths across platforms.

## File Response Types

- Static file: full file or range response.
- Streamed file: incremental reads for very large files.

## Platform Strategies

Linux:

- Prefer `sendfile` for file-backed responses.
- Use `io_uring` with fixed buffers for async reads + socket writes when enabled.
- Use `pread` into pooled buffers as fallback.

macOS/BSD:

- Prefer `sendfile` where supported.
- Fallback to `pread` into pooled buffers.

Windows:

- Prefer `TransmitFile` or `WSASend` with file handles.
- Fallback to `ReadFile` into pooled buffers.

## Buffer Integration

- Zero-copy paths must bypass buffer pools when the OS supports direct send.
- When fallback is required, buffers come from the shared pool and are returned immediately after write completion.

## Range Requests

- Support single-range requests; multi-range may be deferred.
- Validate range bounds before scheduling I/O.
- Ensure range responses respect `max_body_bytes`.

## Caching and Metadata

- Cache file metadata (size, mtime) for hot paths.
- Use `etag`/`last-modified` to enable conditional requests.

## Backpressure

- File I/O counts toward write buffering limits.
- If write backpressure triggers, pause file reads and resume after drain.

## Validation

- Integration tests for small, large, and range responses.
- Performance tests comparing zero-copy vs fallback paths.
