# HTTP QUERY Method (RFC 10008)

## Scope

First-class support for the HTTP QUERY method (RFC 10008, June 2026, Proposed
Standard; the first new HTTP method since PATCH in 2010) across HTTP/1.1, h2,
and h3, in the app router, the reverse proxy, and the gateway response cache.

QUERY is a safe, idempotent, cacheable method whose request semantics live in
the request body instead of the URL. It is "GET with a body, done right":
no URL length limits, query data stays out of URI logs, and responses are
cacheable when the cache key incorporates the request content.

The differentiating feature for swerver as a gateway is **body-keyed QUERY
caching**: caching POST-shaped search/query traffic at the edge, which
mainstream gateways do not do today.

## RFC requirements that drive the design

- Safe and idempotent (RFC 10008 sec 2, 2.6): intermediaries may retry QUERY
  after connection failure; QUERY must never be treated as mutating.
- Cacheable (sec 2.7): "The cache key for a QUERY request MUST incorporate
  the request content and related metadata." Content normalization for the
  key is a MAY, not a MUST.
- Content-Type (sec 2): "Servers MUST fail the request if the Content-Type
  request field is missing or is inconsistent with the request content."
  Missing media type -> 400; unsupported -> 415; well-formed but
  unprocessable -> 422.
- Accept-Query response header (sec 3): a Structured Fields list of media
  ranges advertising the query formats a resource supports; applies to every
  URI sharing the same path.
- Redirect-to-GET (sec 2.5): a 303 response means the query result is
  retrievable via GET on the Location URI. Conditional requests apply per
  RFC 9110 sec 13.
- CORS (sec 4): QUERY is not CORS-safelisted; browsers will preflight.

## Current state

`Method.fromStringExtended` already tolerates QUERY as `.OTHER` with
`method_raw` preserved (src/protocol/request.zig:52), the h1 parser derives
body presence from headers rather than method, and the proxy forwards
`getMethodName()` verbatim. So QUERY passes through swerver today, but with
wrong semantics in exactly the places that matter:

1. Rejected QUERY requests (auth/ratelimit/x402/body-schema) do not set
   `close_after_write`, leaving an undrained body that corrupts the next
   pipelined h1 request (dispatch.zig close gates list only POST/PUT/PATCH).
2. The response cache treats QUERY as mutating: every QUERY invalidates the
   cached entry for its path (dispatch.zig:1040 treats "not GET/HEAD" as a
   write), and QUERY responses are never cached.
3. The proxy classifies QUERY as non-idempotent (forward.zig:973 `else =>
   false`), so it refuses safe retries.

There is no centralized method-property classifier anywhere; safety,
idempotency, cacheability, and body-bearing are open-coded as scattered
`method == .POST or ...` chains. That is the root cause of all three bugs
above and the first thing this spec fixes.

## Design

### Phase 1: method classification core

Add `QUERY` to `request.Method` (src/protocol/request.zig:23), placed before
`OTHER` so the sentinel stays last, plus the `fromString` map entry.

Add property classifiers on `Method` and make them the single source of
truth:

```zig
pub fn isSafe(m: Method) bool          // GET, HEAD, OPTIONS, TRACE, QUERY
pub fn isIdempotent(m: Method) bool    // safe + PUT, DELETE
pub fn isCacheable(m: Method) bool     // GET, HEAD, QUERY
pub fn allowsRequestBody(m: Method) bool // POST, PUT, PATCH, OPTIONS, QUERY, OTHER
```

`OTHER` is conservatively unsafe, non-idempotent, non-cacheable, and
body-allowed. OPTIONS counts as body-bearing: RFC 9110 permits OPTIONS
content, and the eager-arena heuristic already treated it that way. TRACE
(MUST NOT have content) and CONNECT (tunnel; rejected 501 earlier) do not.

Rewire every open-coded site through these helpers:

| Site | Today | Becomes |
|---|---|---|
| dispatch.zig:1990, 1998, 2040, 2052, 2076 close-on-reject gates | `== .POST or .PUT or .PATCH` | `allowsRequestBody()` |
| dispatch.zig:1241, 1379, 2295 + server/http2.zig:1039 eager-arena gates | `!= .GET and != .HEAD and != .DELETE` | `allowsRequestBody()` |
| dispatch.zig:1040 cache invalidation | `!= .GET and != .HEAD` | `!isSafe()` |
| proxy/forward.zig:973 isIdempotent | local switch, `else => false` | `Method.isIdempotent()` |

WASM ABI: `methodCode` (src/wasm/filter.zig:151) gets `.QUERY => 9` as a new
stable code; mirror the constant in the abi.zig SDK. The switch is exhaustive
so the compiler forces this assignment.

Router: add `query()` registration helpers to `Router` and `GroupBuilder`
alongside get/post/put/delete/patch. No implicit mapping between QUERY and
GET routes in either direction (unlike HEAD-implies-GET): the two methods
have different handler contracts (body vs no body).

No h2/h3 protocol work: QUERY is absent from the HPACK and QPACK static
tables by design (they are RFC-frozen) and encodes as a literal `:method`.
Both decode paths already resolve arbitrary method tokens through
`fromStringExtended`.

### Phase 2: dispatch correctness (ships with Phase 1)

The close-on-reject rewiring above is a correctness fix independent of
QUERY: today a body-schema 400 or a rate-limit 429 on any body-bearing
request that is not POST/PUT/PATCH leaves the h1 pipeline poisoned. The
Phase 1 refactor closes it for QUERY and OTHER alike. Regression test: two
pipelined h1 requests where the first is a rejected QUERY with a body; the
second must still parse.

### Phase 3: body-keyed response cache

The headline feature. RFC 10008 requires the cache key to incorporate the
request content and related metadata; swerver's `computeKey`
(src/proxy/cache.zig:358) currently hashes only path + configured vary
headers, and `lookup` hard-rejects non-GET.

Key construction for QUERY entries folds into the existing Wyhash key:

1. the method tag (so a QUERY entry can never satisfy a GET, and vice versa;
   this also future-proofs the key for any other cacheable method),
2. the full raw request body bytes,
3. the request Content-Type value (the "related metadata" the RFC names;
   two identical byte-bodies under different media types are different
   queries).

No normalization in v1. The RFC makes normalization a MAY and warns that
incorrect normalization can serve wrong responses (sec 4); byte-exact keying
is always correct, merely less hit-efficient. Revisit per-media-type
normalization (e.g. JSON key ordering) only with real traffic evidence.

Gates:

- `lookup` (cache.zig:84) accepts `.QUERY` in addition to `.GET`; the
  dispatch pre-forward lookup (dispatch.zig:2145 and the h2 analog) runs
  after body accumulation for QUERY so the body is available for keying.
- Store gate (dispatch.zig:1024) becomes `isCacheable() and status == 200`.
- Invalidation becomes `!isSafe()` (Phase 1), so QUERY stops wiping cached
  GET entries for its path.

Bounds: QUERY caching applies only when the body was fully accumulated under
the route's existing `BodyPolicy` limits; a body that exceeded the
accumulate cap is forwarded uncached. Cached QUERY entries share the
existing per-entry size and TTL limits; no new config keys in v1 beyond the
route-level `cache` block already gating GET.

### Phase 4: proxy retries and origin conformance

- Retries: with `isIdempotent()` fixed, the safety gate passes. Add
  `"QUERY"` to the default `retry_methods` list (upstream.zig:274, currently
  GET/HEAD/OPTIONS); the proxy buffers request bodies, so replaying a QUERY
  body on a fresh upstream connection is mechanical.
- Content-Type guard: for app-router QUERY routes, dispatch rejects a QUERY
  whose body is non-empty but has no Content-Type with 400, per the RFC
  MUST. For proxy routes this is the origin's obligation; the gateway passes
  through (but the missing header also removes the metadata component of the
  cache key, which the keying handles by hashing the absent value as empty).
- Accept-Query: v1 provides a response-header constant and leaves emission
  to handlers/origins. Auto-advertising per route implies knowing each
  route's supported query media types, which the router does not model;
  defer until there is a body-schema integration to derive it from.
- CORS: QUERY is not safelisted; operators who want cross-origin QUERY add
  it to the `cors_allow_methods` config string (a plain string, no schema
  change).

### Out of scope for v1

- Cache-key normalization (MAY; correctness risk outweighs hit-rate gain).
- 303 redirect-to-GET synthesis and conditional QUERY evaluation at the
  gateway: both pass through to the origin unmodified, which is conformant.
- The pre-existing h2 `method_raw` drop (protocol/http2.zig DecodedRequestBlock
  has no method_raw field, so `.OTHER` methods from h2 clients forward with
  an empty token). Unrelated to QUERY once QUERY is first-class; track as
  its own fix.
- Admin API stays GET/POST-only (405 for QUERY is correct there).

## Testing

- Unit: Method classifier truth table; `fromString("QUERY")`; Allow-header
  includes QUERY for a registered route; `methodCode` ABI stability.
- Cache: same path + different QUERY bodies -> distinct entries; same body +
  different Content-Type -> distinct entries; QUERY entry never satisfies
  GET on the same path; QUERY does not invalidate a cached GET; store/hit
  round-trip over h1 and h2.
- Pipeline regression: rejected QUERY (auth 401, ratelimit 429, body-schema
  400) closes the connection; pipelined follow-up on a non-rejected QUERY
  parses cleanly.
- Proxy: QUERY forwarded verbatim over h1 upstream; retry on upstream
  connection failure replays the body once.
- WASM: filter sees method code 9; park/resume path unaffected.
- E2E: k6 scenario driving QUERY with a JSON body through a cached proxy
  route, asserting hit-rate > 0 and byte-identical responses.

## Rollout

Target v0.1.0-alpha.26 as the headline feature (alpha.25 is staged in PR #32
and frozen for release after the tinylaunch window). Phases 1+2 are one PR
(mechanical + correctness), Phase 3 a second (cache), Phase 4 a third
(proxy/conformance). Docs: a guide page with the curl/k6 examples and the
cache-key semantics; update 7.0-rfc-compliance-index.md with RFC 10008
rows.
