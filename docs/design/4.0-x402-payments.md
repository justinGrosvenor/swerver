# x402 Payments (HTTP 402 Flow)

Reference architecture: [2.0-architecture.md](2.0-architecture.md).
Reference requirements: [1.2-requirements.md](1.2-requirements.md).

## Scope

Defines the HTTP 402 payment flow for x402 (protocol v2) and how the server signals payment requirements and accepts payment payloads.

## References

- Protocol spec v2: https://github.com/coinbase/x402/blob/main/specs/x402-specification-v2.md
- HTTP transport v2: https://github.com/coinbase/x402/blob/main/specs/transports-v2/http.md

## Flow Summary

1. Client requests a protected resource.
2. Server replies with HTTP 402 and a `PAYMENT-REQUIRED` header containing base64-encoded `PaymentRequired`.
3. Client retries with `PAYMENT-SIGNATURE` header containing base64-encoded `PaymentPayload`.
4. Server verifies the payload (directly or via facilitator) and serves the resource.

## HTTP Signaling

### Payment Required

- Status: `402 Payment Required`
- Header: `PAYMENT-REQUIRED` (base64-encoded `PaymentRequired` JSON)
- Body: optional; error detail is carried in the header payload.

### Payment Payload

- Header: `PAYMENT-SIGNATURE` (base64-encoded `PaymentPayload` JSON)
- If valid, proceed with normal response.
- If invalid or missing, return 402 with an updated `PAYMENT-REQUIRED` payload.

## Constraints

- The server must not log payment signatures or secrets.
- The payment flow must not require extra out-of-band HTTP requests beyond the 402 retry.
- Payment requirements must include `x402Version: 2` and at least one accepted scheme.

## Integration Approach

- Implement as optional middleware at the router layer (not in core runtime).
- Middleware decides per-route whether payment is required.
- Middleware can cache payment validations (short TTL) to reduce overhead.

## Validation

- Unit tests for correct 402 responses and header encoding.
- Integration tests for request → 402 → paid retry flow.
