# Proxy Response Streaming

Status: scoping (2026-06-11). Owner: proxy subsystem.

## Context

The reverse proxy reads the **entire** upstream response before forwarding
any byte to the client (`proxy.zig readUpstreamResponse`). Commit `590ae46`
fixed the correctness ceiling (responses no longer fail above 64KB; they
grow into a heap buffer bounded by `route.max_response_bytes`, default
32MB), but three structural costs remain:

1. **TTFB** — the client waits for the full upstream transfer before the
   first byte arrives. For a 32MB response from a slow origin this is the
   whole transfer time.
2. **Memory** — N concurrent large responses hold N × size heap buffers.
3. **Reactor blocking** — the deeper problem. All upstream I/O
   (`connectBlocking`/`sendAll`/`recvBlocking`) runs synchronously on the
   reactor thread inside `handleRead`. A slow upstream stalls *every*
   connection on that worker for the duration. Streaming with blocking I/O
   would make this *worse* (the stall would stretch across the entire
   relay), so *streaming is inseparable from making upstream I/O
   event-driven*. This scoping treats them as one effort.

## Goals

- Forward upstream response headers to the client as soon as they parse;
  relay body bytes as they arrive with a bounded per-connection window.
- Upstream sockets become event-loop citizens: no reactor-blocking waits
  on connect, send, or read.
- Bounded memory per relay (window = a few pool buffers, not the body).
- Keep the buffered path for small responses (better syscall economy —
  one writev for header+body — and required for retry/cache/compress/x402
  semantics). nginx's `proxy_buffering` threshold model is the precedent.

## Non-goals (this effort)

- Request-body streaming upstream (exists separately via `handleWithBody`).
- Chunk-wise gzip of streamed responses (streamed responses skip
  `maybeCompress`; revisit later).
- Caching of streamed responses (streamed responses bypass the response
  cache; entries above the cache's size cap were never cacheable anyway).
- WebSocket/upgrade tunnels (already separately implemented).

## Architectural survey (what exists to build on)

- **Event-loop id space**: readiness backends (epoll, kqueue,
  io_uring_poll) are id-agnostic — `registerConnection(conn_id, fd)`
  stores an opaque u64 echoed back on events (`epoll.zig:86-92`). An
  upstream fd can join the loop with a tagged id:
  `UPSTREAM_ID_BIT | client_conn_index`. Zero changes to these backends.
  Reserved ids already follow this pattern (`io.zig` `UDP_SOCKET_ID`).
- **Native io_uring backend** is role-typed (`packUserData(.recvmsg, ...)`,
  multishot recv arming per role) and needs a real `.upstream` role +
  arming path. Deferred to phase 3; until then the native backend keeps
  the buffered path (selection per-route at runtime, and
  `SWERVER_BACKEND=epoll` allows full-streaming A/B on Linux).
- **Backpressure**: `write_buffered_bytes` + the `Backpressure` high/low
  water machinery already pause/resume *client* reads. The relay reuses
  the same accounting to pause *upstream* reads when the client's write
  queue is above high-water.
- **Write side**: `enqueueWrite` + `handleWrite`'s drain loop already
  handle incremental body emission (`pending_body`, `streamBodyChunks`).
  A relay queues body chunks the same way; no new client-side write
  machinery.
- **Timeouts**: `route.timeouts` already has `connect_ms`, `send_ms`,
  `read_ms` (becomes idle-between-bytes), `total_ms`. The connection
  timeout wheel (`setTimeoutPhase`) needs one new phase: `.proxy_relay`.

## Design sketch

### Relay state (new, on Connection or a side table indexed by conn)

```
UpstreamRelay = struct {
    fd: posix.fd_t,
    state: enum { connecting, sending_request, reading_headers, relaying, draining_keepalive },
    pool_conn: *pool.Conn,            // return to keep-alive pool on clean end
    body_mode: enum { content_length, chunked, close_delimited },
    remaining: u64,                    // content_length mode
    chunk_scanner: ChunkScanner,       // incremental chunked parser state
    header_buf_idx: usize,             // proxy pool buffer for request/headers
    started_ms: u64,                   // total_ms enforcement
    headers_forwarded: bool,           // gates retry eligibility
}
```

### Flow

1. `proxy.startStreamingRelay()` (replaces `handle()` for eligible routes):
   non-blocking connect (`EINPROGRESS`), register upstream fd with tagged
   id, state = `.connecting`. Returns immediately — the client connection
   parks (same mechanism as x402 `.pending`: the pipeline loop already
   breaks on parked states).
2. Writable event on upstream → send request bytes (short-write safe),
   state = `.reading_headers`.
3. Readable events → read into a pool buffer; incremental header parse.
   On complete headers: run the existing `normalizeUpstreamResponse`
   (hop-by-hop strip, Via, route header rules), queue client response
   headers, mark `headers_forwarded`, state = `.relaying`.
4. `.relaying`: each readable event reads up to the window into pool
   buffers and `enqueueWrite`s them to the client (h1: raw for
   content-length; re-emitted chunked framing for chunked upstreams).
   Pause rule: if client `write_buffered_bytes` > high-water, do not read
   upstream (readiness is edge-triggered: record `upstream_readable=true`
   and resume from `handleWrite`'s drain when below low-water).
5. Body end (remaining==0 / final chunk / EOF): release upstream to the
   keep-alive pool (clean) and unpark the client connection (resume
   pipeline). Mid-relay upstream failure: client h1 → connection close
   (the only honest signal after a 200 started); log.
6. Client abort mid-relay: close upstream (do not pool).

### Eligibility / threshold

- Buffered path remains the default for everything else and is the
  fallback whenever streaming is ineligible.
- Streaming engages when ALL hold: route has `stream_responses: true`
  (config flag, default false initially), client protocol is h1 (phase 1),
  backend is a readiness backend (phase 1-2), route has no response
  cache, no x402 settle-park, method not HEAD.
- Optional later: auto-engage when upstream `Content-Length` >
  `stream_threshold_bytes` even without the flag (nginx-style), once
  proven.

### Semantics that change (and must be documented)

- **Retry**: impossible once `headers_forwarded`. Before that point the
  existing retry ladder still applies (connect/send failures, 5xx-retry
  happens at header time before forwarding).
- **maybeCompress / response cache / mirror**: skipped for streamed
  responses (mirror fires on headers with null body, as handleWithBody
  does today).
- **Failure mid-stream**: truncation surfaces as h1 connection close
  (or h2 RST_STREAM in phase 4) — never a fabricated 502 after bytes
  have been sent.

## Phases, effort, risk

| Phase | Scope | Effort | Risk |
|---|---|---|---|
| 0 (done, `590ae46`) | Bounded buffering to 32MB; correctness | — | shipped |
| 1 | Streaming relay: h1 client ↔ h1 upstream, readiness backends, flag-gated per route, buffered fallback intact | ~1-1.5 wk | medium — new state machine, but additive behind a flag; e2e harness exists (`test-proxy-large.sh` pattern + slow-upstream/backpressure cases) |
| 2 | Backpressure hardening + soak: slow-client + fast-upstream, slow-upstream + pipelined client, abort matrices, timeout phases | ~0.5 wk | low — test-heavy |
| 3 | Native io_uring backend: `.upstream` role in user_data packing, recv arming, writable arming | ~1 wk | medium-high — touches the perf-critical backend; gate + A/B vs epoll |
| 4 | h2/h3 client side: relay into DATA frames with h2 flow control / h3 stream writes; TLS upstreams (async handshake state) | ~1-2 wk | high — flow-control interaction; do after 1-3 are boring |

Recommendation: build phase 1 next time proxy work is scheduled; phases
3-4 only if a workload actually needs them (the current platform's edge
responses are asset-sized — phase 0's bounded buffering already serves it
correctly; phase 1's wins there are TTFB on cold large assets and worker
isolation from slow origins).

## Test plan (phase 1 acceptance)

- Extend `scripts/test-proxy-large.sh`: streamed route variant of every
  existing check (byte integrity 10KB-10MB, CL + chunked, keep-alive).
- New: slow upstream (drip 64KB/100ms) — assert client TTFB < 200ms while
  total transfer completes intact; assert OTHER clients on the same worker
  get sub-ms responses during the drip (the reactor-isolation proof).
- New: slow client (curl --limit-rate) + fast upstream — assert bounded
  memory (RSS) and upstream pausing.
- Abort matrix: client disconnect mid-relay; upstream reset mid-relay;
  timeout in each state.

## Open questions

1. Park mechanism: reuse the x402 `.pending` conn state or add a distinct
   `.proxy_streaming` state? (Leaning distinct — x402 has settle
   semantics tangled in.)
2. Chunked re-emission vs verbatim relay for h1 clients: verbatim is
   cheaper but forwards upstream's trailer/extension quirks; re-emission
   normalizes. (Leaning re-emission via existing chunk writer.)
3. Upstream TLS (https origins): currently unsupported in the proxy at
   all — phase 4 prerequisite, separate scoping.
4. Should `max_response_bytes` still bound streamed responses? (Leaning
   no — streaming exists to remove the bound; keep `total_ms` as the
   safety. Flag-off routes keep the cap.)
