# Core Runtime: Event Loop Integration

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference architecture: [2.0-architecture.md](2.0-architecture.md).

## Scope

Defines how the runtime consumes platform I/O events and drives connection state, buffers, and protocol stacks without per-request allocations.

## Event Model

Runtime consumes canonical events:

- accept(handle)
- read(conn_id, bytes)
- write(conn_id, bytes)
- error(conn_id, code)
- datagram(conn_id, bytes): UDP packets for QUIC/HTTP3

Platform backends must map OS events into these canonical events.

## I/O Backends

The runtime selects one of four backends at startup:

- `bsd_kqueue`: kqueue (macOS/BSD).
- `linux_epoll`: epoll (Linux fallback).
- `linux_io_uring_poll`: io_uring in poll/readiness mode (Linux).
- `linux_io_uring_native`: io_uring completion model with kernel-delivered reads, provided buffers, and a threaded multishot accept (Linux, default when probed).

## Poll Loop

High-level loop:

```
poll() -> [events]
for event in events:
  lookup connection by conn_id
  dispatch to handler
  update last_active_ms on progress
  enforce backpressure
  enforce timeouts
```

Rules:

- If no events, block with a bounded timeout to avoid busy-wait.
- Events for closed connections are discarded.
- Read events can be coalesced; write events should report bytes completed.

## Accept Flow

A worker binds and registers **every** configured listener (`server.cfg.listeners[]`), not just one. There are two accept delivery models:

- **Readiness** (kqueue, epoll, io_uring_poll): a readiness event fires for a listener. The backends tag all listener fds identically, so `handleAccept` drains *every* bound listener fd, calling `accept4()` in a userspace loop until `WouldBlock` (non-ready listeners return immediately).
- **Completion** (io_uring_native): a dedicated acceptor thread arms a multishot accept on its own ring and hands the already-accepted client fd inline. The dispatcher routes it straight to `handlePreAccepted`, with no `accept4()` syscall.

Per-connection setup is the same for both models:

- Acquire a connection slot.
- Assign buffers if available; otherwise defer accept or close immediately.
- Initialize protocol selection (cleartext or TLS handshake).
- Assign `conn_id` after slot acquisition and register it with the backend.

## Read Flow

- If backpressure prohibits reads, stop scheduling read events.
- On bytes read, pass slices to protocol parser.
- Protocol outputs response metadata for the write path.

## Write Flow

- Writes are queued in buffer handles or sendfile descriptors.
- When write completes, release buffers and update `write_buffered_bytes`.

## Error Flow

- Map backend error to canonical categories.
- Transition to `error`, notify protocol stack, and schedule close.

## Scheduling Model

- Single-threaded per-core loops are preferred for cache locality.
- Accept fan-out uses `SO_REUSEPORT` on supported platforms.
- Connection ownership is pinned to a worker; handoff requires explicit state transfer.

## Validation

- Integration tests ensure no busy-waiting in empty poll loops.
- Stress tests validate no event loss under high concurrency.
