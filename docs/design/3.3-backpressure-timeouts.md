# Core Runtime: Backpressure and Timeouts

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference invariants: [1.3-invariants.md](1.3-invariants.md).

## Scope

Defines how the runtime limits read/write buffering and enforces timeouts across protocols.

## Backpressure Model

Each connection tracks:

- `read_buffered_bytes`
- `write_buffered_bytes`

Thresholds:

- `read_high_water` / `read_low_water`
- `write_high_water` / `write_low_water`

Rules:

- If `read_buffered_bytes >= read_high_water`, do not schedule further reads.
- If `read_buffered_bytes <= read_low_water`, resume reads.
- If `write_buffered_bytes >= write_high_water`, do not queue more writes.
- If `write_buffered_bytes <= write_low_water`, resume writes.
- Backpressure applies before parsing to avoid allocating work the system cannot flush.

## Timeout Phases

Phase is driven by protocol state:

- `idle`: no in-flight request.
- `header`: request line and headers being received.
- `body`: request body in progress.
- `write`: response queued or in-flight.

Rules:

- Timeouts are based on `last_active_ms`, updated only on progress.
- A timeout must transition the connection to `draining` or `error` depending on protocol rules.
- Timeouts are enforced even when TLS is enabled.

## Idle Timeout

- Idle connections should be closed after `idle_ms`.
- For HTTP/2 and HTTP/3, idle applies to the connection, not to streams.

## Header Timeout

- Timer starts when a new request begins (first byte observed).
- If headers exceed time, respond with `408` (HTTP/1.1) or stream error (HTTP/2/3).

## Body Timeout

- Enforced per request for HTTP/1.1.
- For HTTP/2/3, enforced per stream.

## Write Timeout

- Start timer when write is first queued.
- If exceeded, close the connection after attempting to flush any in-flight writes.

## Validation

- Unit tests for timeout thresholds.
- Integration tests that simulate slowloris (header/body slow reads) and slow writer clients.
