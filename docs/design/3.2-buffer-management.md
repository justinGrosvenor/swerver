# Core Runtime: Buffer Management

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference invariants: [1.3-invariants.md](1.3-invariants.md).

## Scope

Defines buffer allocation, ownership, and reuse for all read/write paths. Designed for zero per-request heap allocations.

## Buffer Pool

The runtime owns a fixed-size pool:

```
BufferPool {
  buffer_size: usize
  buffer_count: usize
  storage: [buffer_count][buffer_size]u8
  free_stack: [buffer_count]u32
}
```

Rules:

- Buffers are fixed-size slabs; no heap growth at runtime.
- Pool exhaustion is treated as backpressure and must pause further reads.
- Buffer handles are opaque to protocol stacks; only slices are shared.

## Read Path

- On accept, allocate a read buffer if available.
- All parse operations use slices of read buffers (zero-copy).
- If a request spans buffers, the parser must support multi-segment inputs without compaction.
- If limits are exceeded, reject with `431` or `413` without copying.

## Write Path

- Response headers and small bodies should coalesce into a single buffer.
- Large bodies use zero-copy sendfile/async file APIs when available.
- For streaming responses, maintain a write queue with buffer handles; enforce write high-water.

## Ownership Rules

- A buffer handle can be owned by at most one connection at a time.
- A buffer slice cannot outlive the buffer handle.
- Protocol code cannot retain slices across state machine boundaries unless the runtime pins the buffer.

## Header Index Tables

- Parsed headers are tracked via a fixed-size index table of `(name_slice, value_slice)` pairs.
- The table is preallocated per connection and reused across requests.
- If the header count exceeds the table capacity, reject with `431`.
 - Table capacity is configurable and defaults to 128 headers.

## Buffer Sizing

- Default `buffer_size` should cover typical HTTP/1.1 headers + small body.
- `buffer_count` should cover `max_connections` plus expected in-flight writes.
- Backpressure thresholds should be a multiple of `buffer_size`.

## Error Handling

- Any error path must release all buffers held by the connection.
- Double-release is an invariant violation; guard in debug builds.

## Validation

- Unit tests for pool acquire/release behavior.
- Stress tests to validate no leaks under repeated accept/close cycles.
