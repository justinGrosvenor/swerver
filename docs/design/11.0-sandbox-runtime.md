# Sandbox Runtime (Nether Integration)

Status: scoping (2026-06-21). Owner: sandbox subsystem (new). This is a
long-running, co-evolving effort: the design below is the starting point,
and the protocol/ergonomics are expected to change as swerver and Nether
iterate against each other. The spike uses a subprocess + unix control
socket transport (available today, zero Nether changes); the in-process
embeddable core is the documented end-state. Lives on the
`nether-sandbox` branch alongside the code it specs.

## Context

swerver runs Tier-1 cheap logic in-process (design 10.0, WASM edge
functions). Tier-2 is heavy, hostile, arbitrary agent code in hardware
isolation: that is [Nether](https://github.com/justinGrosvenor/nether),
the microVM layer. This document is how swerver *drives* Tier-2: route an
HTTP request to a sandboxed guest and return its result, behind the same
gateway and meter.

Nether has crossed its done-line. It PVH-boots Linux to an interactive
shell, runs an exec-over-vsock agent, snapshot-forks (copy-on-write), and
enforces govern axes (egress firewall, bandwidth, runtime budget) plus
egress metering. Its own roadmap states the remaining work is "the layer
up": *"the VMM now has boot, SMP, snapshot/fork, vsock control, and real
networking; the platform integration is the next layer up."* That layer
is this document. Nether already exposes the exact seam it needs: a
per-sandbox unix **control socket** plus a `nether.conf` per sandbox.

The integration is, structurally, **the PostgreSQL client again** (design
9.0). The same three primitives carry it: park-and-resume (suspend the
HTTP request while the guest runs, resume on completion), external-FD
reactor dispatch (`EXTERNAL_ID_BIT`), and a per-worker slot pool. Nether's
control protocol is *simpler* than the Postgres wire protocol: write a
line, read a delimiter-framed reply. So the bulk of this is a new client
module mirroring `db/pg/client.zig`, not new runtime machinery.

Why a long-running branch rather than a one-shot build: the protocol and
lifecycle are the parts that must co-evolve with Nether (exec-REPL grows
into request/response framing; subprocess transport becomes the in-process
embeddable core). The handler-facing API (`ctx.sandbox.exec`) stays stable
while those swap underneath, exactly as the PG client's transport went
plaintext to TLS under a fixed handler API.

## Goals

- Drive a Nether sandbox from a swerver handler: `ctx.sandbox.exec(...)`
  parks the request, the command runs in the guest, the continuation
  resumes with stdout/stderr + exit code. Reuse the design-9.0
  park-and-resume contract wholesale.
- Per-worker sandbox pool mirroring the PG slot pool. Per-VM-per-worker
  ownership falls out for free (Nether's own D3 concurrency answer).
- A stable client API behind a swappable **transport seam**, so the v1
  subprocess + control-socket transport can become the in-process
  embeddable core later without touching a single handler.
- Multi-owner external-FD dispatch so PG, Nether, and (later) proxy
  streaming coexist as reactor citizens.
- Zero per-request heap allocation on the steady-state path (slot recv
  buffers, the same discipline as PG).

## Non-goals (this effort)

- The in-guest runtime, agent, or guest image. That is Nether plus the
  guest rootfs.
- The in-process embeddable core in v1. Subprocess + control socket
  first; the embeddable core is a later phase, documented as the
  end-state so the seam is built for it now.
- Request/response-over-vsock framing in v1. Line-exec first; the
  HTTP-shaped protocol is co-designed with Nether (open question 1).
- Cross-worker sandbox sharing. Per-worker, like PG.
- Snapshot, fork, and migration *mechanics*. Nether owns those; swerver
  drives them (open question 2).
- A subprocess supervisor at fleet scale. The host provisioning is the
  `ansible-swerver` `kvm_host` concern, separate.

## Alternatives considered (2026-06-21)

**HTTP-proxy to a sandbox listening on a TCP port.** Rejected. It gives
the guest a network path, which defeats the isolation premise (Nether's
egress firewall exists precisely so the guest has no ambient network), and
it bypasses the control channel that carries govern/meter. The guest must
be reachable only over vsock/control, never a socket the guest opened.

**One long-lived sandbox per worker, shared across requests.** Rejected
for untrusted multi-tenant: one request's guest state (files, memory,
processes) leaks into the next. The model is per-request or per-tenant
sandboxes drawn from a warm pool, cheap because of snapshot-fork.

**Drive Nether over its stdio (the agent REPL on stdin).** Rejected:
single global stdio, not multiplexable across sandboxes. The control
socket is the per-sandbox programmatic API Nether built for exactly this
("the platform spawns nether per sandbox and attaches to its control
socket").

**Subprocess transport vs the in-process embeddable core.** Subprocess
now: available today with no Nether changes, and a separate address space
is a *feature* during bring-up (a crashing VMM cannot take the worker
down). Embeddable core later: no IPC hop, device eventfds register into
`IoRuntime`, vsock rides park-and-resume directly; the thesis end-state.
The client API hides which is in use.

## Architectural survey (what exists to build on)

**Nether side (the control protocol swerver speaks):**
- A per-sandbox **unix-domain control socket**, path from `nether.conf`
  `control_socket=`. Line-oriented: write `<command>\n`, the command runs
  in the guest via `/bin/sh`, and the reply streams back framed as
  `<stdout+stderr bytes>0x1e<exit-code>\n`.
- Host-intercepted verbs (also line commands): `__stats__` (metering
  report), `__shutdown__` (clean teardown, replies `OK shutting down\n`),
  `__put__ <args>` / `__get__ <args>` (file transfer, payload over vsock
  with length framing, not the line socket).
- One client at a time per socket. `nether.conf` also carries `cpus`,
  `ram_mb`, and `restore_from=<base-snapshot>`.

**swerver side (all present on main, all PG-proven):**
- `io.zig`: `EXTERNAL_ID_BIT`, `isExternalId`, `registerExternalFd` /
  `armExternalWritable` / `unregisterExternalFd`.
- `db/pg/handler_api.zig`: the park-and-resume contract (`Continuation`,
  `ResumeContext`, typed stash, generation check, debug poisoning of
  borrowed result memory).
- `db/pg/client.zig`: per-worker slot pool, park table, frame-scan +
  `deliverOutcome` + `failSlot`, `tick`, `installResume`.
- `server/dispatch.zig`: the external-FD branch (`isExternalId` ->
  owner.onEvent), `pgResume` hook, housekeeping `tick`.
- `connection.zig`: `.db_parked` connection state (one enum byte).
- `config.zig` / `config_file.zig`: the optional-feature config-block
  pattern (`postgres`).

## Design sketch

### NetherClient (`src/sandbox/nether.zig`)

Per-worker pool of sandbox slots. Each slot owns:
- a spawned `nether` subprocess (or, later, an in-process VM),
- its connected control-socket fd, registered via `registerExternalFd`,
- a state machine: `spawning -> connecting -> ready -> busy -> failed`,
- a park table entry while busy (the awaiting HTTP request).

`exec(conn_index, conn_id, command, stash, continuation)` writes
`command\n` to the slot, parks the request, returns `Response.parked`.
`onEvent(slot, kind)` on readable accumulates the reply into the slot recv
buffer, scans for the `0x1e<exit>\n` trailer, and on a complete frame
calls `deliverOutcome` with a `SandboxResult`. This is `db/pg/client.zig`
with a one-delimiter protocol in place of the Postgres frame parser; the
slot/park/generation-check/failSlot scaffolding is identical.

### Handler API (`src/sandbox/handler_api.zig`)

```zig
fn runUname(ctx: *HandlerContext) Response {
    return ctx.sandbox.exec("uname -srm", Stash, .{}, onResult)
        catch Response.serviceUnavailable();
}
fn onResult(rctx: *SandboxResumeContext) Response {
    const r = rctx.result catch return Response.serverError();
    // r.stdout / r.stderr borrow the slot recv buffer (continuation-scoped);
    // r.exit_code is the guest command's status.
    ...
}
```

`SandboxResult { stdout: []const u8, stderr: []const u8, exit_code: i32 }`,
borrow-only like `pg.Result`. The `SandboxResumeContext` mirrors
`pg.ResumeContext` (no request field; typed stash; repark for chains).

### Transport seam

A minimal interface the client talks through, so the transport can change
without touching handlers or the client's park logic:
`open(conf) -> fd`, `send(bytes)`, `recv(buf) -> n`, `close()`. v1 is the
unix control socket; the embeddable-core phase swaps in an in-process vsock
endpoint that drives park-and-resume directly. Keeping this seam clean is
the single most important structural constraint of the branch.

### Lifecycle

Spawn `nether` with a generated `nether.conf` (a `control_socket` path
unique per slot per worker, plus `cpus`/`ram_mb`/`restore_from`), wait for
the socket to appear, connect, register the fd. On a snapshot-fork the
guest's vsock state is not preserved (known Nether gap), so the control
connection reconnects per fork. `tick` enforces an exec deadline and
respawns failed slots with backoff (PG `failSlot`/retry pattern).

### Multi-owner external-FD dispatch

Today `isExternalId(conn_id) -> pg_client.onEvent`. With a second owner,
partition the `EXTERNAL_ID_BIT` low-32 slot space by owner range (or add a
small owner tag), and route each tagged event to PG / Nether / proxy
accordingly. Small shared-infra change; it pays off three times (PG,
Nether, proxy streaming), which is the signal it is the right seam.

### Connection state and govern/meter

A `.sandbox_parked` value on the connection state enum (one byte, exactly
like `.db_parked`). `__stats__` surfaces per-sandbox metering; feeding it
into the x402 meter (per-request settlement for sandbox compute/egress) is
the platform tie-in (open question 5).

## Phases, effort, risk

| Phase | Scope | Effort | Risk |
|---|---|---|---|
| 0 | Exec vertical slice: subprocess + control socket, line-exec, one route, park -> guest -> framed reply -> continuation. The spike. | S-M | Low-med (subprocess lifecycle) |
| 1 | Sandbox pool + lifecycle: spawn/connect/respawn, nether.conf generation, exec deadline, generation check, multi-owner external-FD dispatch. | M | Med |
| 2 | Request/response protocol: HTTP-shaped framing over the channel (method/path/headers/body in, status/headers/body out), co-designed with Nether. | M-L | Med-high (co-evolution) |
| 3 | Warm pool / clone-per-request: snapshot-fork driven, sub-100ms cold start, reconnect-after-fork. | M-L | Med |
| 4 | Embeddable in-process core: device eventfds -> IoRuntime, vsock over park-and-resume, drop the subprocess and unix socket. | L | High (the end-state) |

A useful slice (phases 0-2) is the bulk of the value; phases 3-4 are the
edge-product and the thesis end-state. The client API is fixed from phase
0; everything after changes transport and protocol beneath it.

## Test plan

- **Phase 0 (live):** route -> real `nether` subprocess -> guest runs the
  command -> output returns as the HTTP body, exit code as a header.
  Requires a KVM/HVF box with a guest image.
- **Frame parser (DB-free unit test):** the `0x1e<exit>\n` scan, including
  a reply split across multiple reads and a trailer split across reads
  (mirror the PG frame-split tests). This is the pure, CI-runnable core.
- **Lifecycle:** sandbox crash mid-exec -> respawn; client disconnect
  while parked -> generation check drops the resume; exec deadline fires.
- **Protocol fuzz:** malformed/garbage reply bytes must never escape the
  slot recv buffer bounds.
- **Host provisioning** is covered by `ansible-swerver` (`kvm_host`).

## Open questions (the co-evolution agenda)

1. **Protocol shape.** Line-exec vs request/response-over-vsock. The
   platform wants "route this HTTP request into a sandboxed handler, get an
   HTTP response back," which is richer than `echo | sh`. This is the first
   thing to settle with Nether and it shapes the `SandboxResult` and the
   in-guest agent contract.
2. **Pooling and forking.** Who drives snapshot-fork (swerver asks Nether
   to fork a base, vs pre-spawning N `restore_from` processes)? Warm-pool
   sizing; reconnect-after-fork given vsock state is not snapshotted.
3. **Transport seam.** The minimal interface that stays stable across
   subprocess -> embeddable core. Get this wrong and phase 4 is a rewrite.
4. **Sandbox identity and lifetime.** Per-request ephemeral vs per-tenant
   warm vs per-session. Ties to the platform's tenancy model.
5. **Govern/meter integration.** How `__stats__` and egress metering feed
   the x402 per-request meter.
6. **Failure semantics.** Guest OOM / timeout / crash -> which HTTP
   status; partial output captured before a crash; idempotency.
7. **Lifecycle ownership.** Does the swerver worker own subprocess
   spawn/reap, or a separate per-host supervisor it attaches to? Affects
   density and crash recovery at scale.
