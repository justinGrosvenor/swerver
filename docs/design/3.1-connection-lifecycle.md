# Core Runtime: Connection Lifecycle

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference invariants: [1.3-invariants.md](1.3-invariants.md).

## Scope

Defines connection identity, state transitions, lifecycle ownership, and safe shutdown semantics. Applies to all transports and protocols.

## Data Model

```
Connection {
  index: u32
  id: u64
  state: State
  state_enter_ms: u64
  last_active_ms: u64
  read_buffered_bytes: usize
  write_buffered_bytes: usize
  read_buffer: BufferHandle?
  write_buffer: BufferHandle?
  peer: SockAddr?
  proto: ProtocolKind
  flags: bitset (half_closed_read, half_closed_write, drain_after_write)
}
```

- `index` is a stable slot in the connection pool.
- `id` is a monotonically increasing identifier used in logs/metrics; never reused across lifetime.
- `last_active_ms` is updated on any read/write progress, not only on accept.
- `proto` starts as `unknown` and is set after TLS/ALPN or cleartext selection.

## State Machine

Allowed transitions:

```
accept -> handshake -> active -> draining -> closed
accept -> active -> draining -> closed
active -> error -> closed
handshake -> error -> closed
accept -> error -> closed
```

Rules:

- Transitions must be explicit and validated; invalid transitions are fatal invariants.
- `draining` means no new requests are accepted; only existing writes are allowed.
- `error` is terminal and must release all buffers before `closed`.

## Ownership and Lifetime

- The connection pool owns the `Connection` memory; it is never freed until shutdown.
- Buffers are owned by the runtime; a connection may hold zero, one, or two buffers.
- Protocol stacks must not retain references to buffers after returning control to the runtime.

## Shutdown Semantics

- On protocol error, transition to `error`, schedule connection close, and emit proper protocol error codes.
- On client-initiated close, mark half-closed; drain pending writes and then close.
- On timeout, transition to `draining` or `error` based on phase (see [3.3-backpressure-timeouts.md](3.3-backpressure-timeouts.md)).

## Connection Allocation

- Acquire from a fixed-size pool (`max_connections`).
- If exhausted, immediately refuse accepts (or apply SYN backlog tuning at the OS layer).
- `id` must be generated without contention; per-thread counters are allowed if combined with worker id.

## Edge Cases

- Socket reuse: if OS reuses FD, always map by runtime connection index + generation, not FD alone.
- Spurious events: events for closed connections must be discarded safely.
- Double close: ignore subsequent close events after `closed`.

## Validation

- Unit tests for allowed state transitions and invalid transition rejection.
- Fuzz or property tests to ensure no buffer leaks on error paths.
