# WASM Edge Functions

Status: **Phase 1 implemented** (2026-06-27), behind the `-Denable-wasm` build
flag (off by default; demand-gated). Owner: gateway subsystem. Phase 0's go/no-go
spike resolved the runtime choice in favour of **wasm3** (see Implementation
status below). The two-tier relationship to Nether (Context) is unchanged; this
remains a single compute story with Nether.

## Implementation status (2026-06-27)

Phase 0 (spike) and Phase 1 (sync metadata-filter MVP + hardening) are built and
tested. Everything is gated behind `-Denable-wasm` and does not touch a default
build.

- **Runtime decision: wasm3** (vendored C, `vendor/wasm3/`). The Phase 0 spike
  benchmarked wasm3 vs zware (pure-Zig) against a native baseline: wasm3 ~54
  ns/op vs zware ~167 ns/op (native ~3 ns), both far under the single-digit-µs
  bar. The non-obvious call was wasm3 over the ethos-favoured zware: wasm3
  preserves the single static binary too (proven via static-musl cross-compile),
  is ~3x faster, has a proven and nearly-free fuel patch, and is immune to Zig
  version churn (zware broke on 0.16 and needed a community re-port). Full
  write-up in `spike/wasm/RESULTS.md`.
- **Built:** vendored wasm3 + per-loop-iteration fuel patch (`vendor/wasm3/`,
  `PATCHES.md`); low-level binding (`src/wasm/runtime.zig`); per-worker instance
  pool + custom ABI (`src/wasm/filter.zig`); config→runtime bridge
  (`src/wasm/manager.zig`); per-route integration in BOTH the embedded
  `Router.handle` and the proxy gateway dispatch; the `wasm_filters` config block
  (`src/config_file.zig`); server startup/reload auto-wiring (`src/server.zig`,
  per-worker pools, rebuilt on config reload).
- **Resource bounds enforced:** fuel (compute, per invocation) + linear-memory
  cap (a runaway `memory.grow` returns -1 instead of OOMing the worker). The
  wall-clock deadline is deferred to Phase 3 (parked host calls) because on a
  single-threaded reactor fuel is the binding bound and a tick deadline cannot
  fire while a guest spins (see the note on `DEFAULT_FUEL` in filter.zig).
- **Security tests passing:** runaway-loop → fuel trap → fail closed;
  memory-bomb refused at the cap; ABI fuzz (out-of-bounds ptr/len always traps,
  never escapes linear memory); fail-closed on trap and pool exhaustion.
- **Phase 2a (request-body read) built:** `body_len()` and `read_body(off,
  ptr, cap)` host imports materialize the request body (slice or scattered ->
  linear, the zero-copy tax) into a bounded per-instance view (`MAX_BODY_VIEW`,
  64 KiB) and serve windows on demand. `body_len()` reports the true total so a
  filter can detect a body larger than its view and reject rather than trust a
  truncated prefix. Available on BOTH the embedded Router (where `req.body` is
  populated by the time the filter runs, including the accumulated/scattered
  path) AND the proxy gateway: the proxy carries the accumulated body in a
  separate `BodyView`, which is mapped zero-copy onto a `RequestBody` for the
  filter (the two share the scattered-buffer layout). See
  `proxy.bodyViewToRequestBody`.
- **Phase 3a-core (instance-level park/resume) built:** the lifecycle for an
  async host call, isolated in filter.zig and unit-tested without the reactor.
  A filter calls `host_call(ptr, len)` to stage an outbound request and returns
  the parked code (3); `invokeOutcome` pins the instance (`.parked`, withheld
  from the pool) and surfaces the staged request. When the host call completes,
  `resumeCall(result)` re-enters the guest's `on_resume()` export (which reads
  the result via `read_call_result` and returns a final Decision) and releases
  the instance; `cancelPark` releases without resuming (host-call failure /
  timeout / client disconnect). This is a TWO-CALL model (on_request returns,
  on_resume runs later), because wasm3 has no stackful coroutines; the pinned
  instance's linear memory carries state between the calls (it IS the stash, per
  the design). The synchronous router/proxy pre-hook cannot resume, so a parking
  filter there fails closed (500) until the reactor wiring lands.
- **Not yet built:** Phase 2b (response-body rewrite, which needs a
  response-side filter hook rather than a pre-hook; respond() already sets a
  body on the reject path); Phase 3 reactor wiring (a `.wasm_parked` connection
  state + park table entry pinning the instance, an async outbound transport,
  and a resume on completion, mirroring the PG park-and-resume path); and the
  Phase 3 Nether vsock bridge (blocked on Nether). Still demand-gated.

### Operational contracts and caveats (Phase 1)

- **Filters must be stateless across invocations.** A pooled instance serves
  many requests from different end-users, and its wasm linear memory + globals
  are NOT reset between calls (wasm3 has no cheap re-instantiate; zeroing memory
  per request would defeat the latency budget). Anything a filter writes to a
  global or to linear memory leaks into the next request on that instance. Not a
  sandbox escape (same tenant module), but a real isolation property the author
  owns. See the contract note atop `src/wasm/filter.zig`.
- **The `wasm_filters` config block binds to PROXY routes only** (matched by
  `path_prefix`). The embedded `Router` is built programmatically, not from the
  config file, so embedded-app routes are wired via the Zig API
  (`router.attachWasmFilter` / `wasm.manager.loadAndAttachRouter`), not config.
- **A reject short-circuits before the post-middleware response-header merge**,
  so a filter-rejected response does not carry the chain's CORS/HSTS/security
  headers. This matches existing native-middleware reject semantics (the chain's
  own `.reject` returns immediately too), so it is consistent, not wasm-specific.
- **`log()` output is emitted only in Debug builds.** The message is
  guest-controlled and callable up to the fuel budget; release builds drop the
  output (still trapping on an out-of-bounds pointer) to avoid a stderr-spam
  vector. A later ABI revision can route it to a rate-limited sink.

## Context

swerver runs untrusted code in exactly one place today: nowhere. Gateway
behavior is native Zig middleware plus declarative config. Two separate demands
push past that, and they are NOT the same demand:

1. **Run arbitrary, possibly hostile, agent workloads.** This is heavy compute,
   full syscalls, any language. It is answered by **Nether** (the microVM
   isolation layer): hardware-enforced EPT/IOMMU isolation, "cannot, not won't."
   Cold start is tens of milliseconds even snapshot-forked. This is **Tier 2**.

2. **Run cheap user logic on the request hot path.** Auth transforms, header and
   routing rewrites, request validation, rate-limit/metering keying, small policy
   checks. This runs on *every request* and must cost single-digit microseconds
   inline on the reactor. A microVM physically cannot enter this latency band.
   This is **Tier 1**, and it is what this document specs: **WASM edge
   functions.**

The relationship to Nether is the whole point and resolves what would otherwise
look like overlap. Nether subsumes WASM-*as-isolation-mechanism*: if you have
hardware-isolated microVMs you do not need a bytecode sandbox to safely run heavy
untrusted code. Nether does **not** subsume WASM-*as-cheap-inline-
programmability*: a microVM per request-filter is absurd. The two are not
competitors, they are two altitudes of one spectrum (the Cloudflare
Workers+Containers and Fly shape), behind one gateway and one meter. An edge
function is also the natural *mediator* for a Nether guest: a Tier-1 filter that
authenticates, meters, and routes a request into a Tier-2 microVM over the same
vsock + park-and-resume spine Nether already uses.

**This is demand-gated.** Native middleware and config already cover swerver's
own declarative gateway needs, and native Zig beats a WASM filter for anything
swerver implements itself. Edge functions exist for *user-authored* hot-path
logic. The trigger to build past the Phase 0 spike is an external party wanting
to put their code on the hot path, not internal feature work. Until then this is
a spike and a spec.

**The unifying primitive.** park-and-resume (built and benchmarked for the
Postgres client, design 9.0) is becoming swerver's universal "suspend the
request, resume when an external/untrusted computation answers" mechanism. PG
queries use it. An edge function's outbound host call uses it. Nether's vsock
guest channel uses it. Whatever order these ship in, that investment compounds,
and an edge function calling *into* a Nether guest is the same control flow we
already know works.

## Goals

- Run user-supplied WASM modules as filters on the request hot path, as one more
  entry in the existing middleware chain. Single-digit-microsecond invocation
  overhead for a trivial filter is the bar; if the runtime cannot approach it,
  the feature does not ship (see Phase 0).
- Per-worker instance pool, pre-instantiated, zero per-request heap allocation,
  no cross-worker sharing. Mirrors the buffer pool and the PG slot pool.
- A Decision-returning ABI that maps onto the existing middleware `Decision`
  union: allow, reject(status, body), modify(headers).
- Async host calls (outbound HTTP to an auth/policy service; vsock into a Nether
  guest) via park-and-resume: the filter suspends, its instance is pinned, the
  request parks, the result is delivered on resume.
- Mandatory resource bounding: fuel/op-count interruption, linear-memory cap,
  per-invocation wall-clock deadline. The single-threaded reactor means a
  runaway filter wedges the whole worker, so preemption is structural, not a
  nicety.
- Single-static-binary ethos preserved: an interpreter that vendors or links
  cleanly and survives cross-compilation, not a JIT that drags in a runtime.

## Non-goals (this effort)

- Heavy or arbitrary agent workloads. That is Nether (Tier 2). An edge function
  is a filter, not a process.
- proxy-wasm ecosystem compatibility in v1. A minimal custom ABI ships first;
  proxy-wasm is a later, perf-gated decision (see Open questions).
- Request/response *body transformation* in v1. The zero-copy boundary makes
  body access the expensive path; metadata filtering ships first.
- JIT or near-native execution speed. Interpreter first; revisit only if
  benchmarks demand it.
- WASM as the implementation language for swerver's own gateway features. Native
  Zig wins there every time.
- WASI or a general syscall surface. Host capabilities are explicit, few, and
  granted, not ambient.

## Alternatives considered (2026-06-18)

**Lua (LuaJIT).** The incumbent choice for gateway scripting (nginx, APISIX,
Kong). Faster per call than an interpreted WASM, but single-language, and LuaJIT
is itself a heavy C dependency. It loses the two things that make WASM the right
strategic fit for this platform: polyglot authoring (Rust, Go, AssemblyScript, C)
and a real sandbox. Rejected; this was the original "WASM instead of Lua"
decision.

**Native dynamic plugins (dlopen of `.so`).** Fastest possible, but no sandbox,
ABI-fragile across compiler versions, and it breaks the single-binary story.
Unacceptable for untrusted user code. Rejected.

**"Just use Nether for everything."** Cannot enter the per-request latency band
(microVM cold start is tens of ms vs filter microseconds). This is precisely why
edge functions survive the existence of Nether rather than being replaced by it.
Rejected as a category error: different altitude.

**Runtime: wasmtime (JIT) vs wasm3 / zware (interpreter).** The fork in the road.
wasmtime gives near-native speed and built-in fuel/epoch preemption, but it is a
1-2 MB C++ dependency with its own allocator and threads, and it kills the
cross-compiled static-binary posture (release binaries ship today without even
OpenSSL). wasm3 (~200 KB C interpreter) or zware (pure-Zig) fit the ethos;
interpreter overhead at middleware altitude is probably acceptable but unproven.
Lean interpreter, with the Phase 0 spike as the deciding measurement. Not
resolved here on purpose.

## Architectural survey (what exists to build on)

- **The middleware seam is a pure function boundary.**
  `MiddlewareFn = fn(*Context, RequestView) Decision`
  (`src/middleware/middleware.zig`), invoked from `Router.handle()`
  (`src/router/router.zig`). A WASM filter is one more pre-hook with zero changes
  to the chain. The `Decision` union (allow / reject / modify / skip /
  backpressure) is exactly the ABI return surface.
- **Per-worker pools already exist as the pattern.** `buffer_pool.zig`
  (free-stack + acquired bitmap) and the PG slot pool (`db/pg/client.zig`:
  embedded fixed buffers, acquire/release, a "busy" state that withholds a slot
  from reuse). The WASM instance pool is the same shape, and "busy" maps to
  "pinned across a host-call park."
- **park-and-resume is generalizable** (design 9.0, `db/pg/handler_api.zig`,
  `server/dispatch.zig` pgResume, `runtime/connection.zig` `.db_parked`). The
  contract: handler returns a park sentinel, router skips serialization, the
  request is recorded with a generation check, the reactor continues, completion
  fires a continuation, the response is queued and connection I/O restarts. The
  one wrinkle for WASM (below): the continuation has no request context and the
  256-byte stash is too small for a WASM linear memory, so the *instance* is the
  stash equivalent and gets pinned.
- **External-fd dispatch** (`EXTERNAL_ID_BIT`, `io.zig`): an outbound host-call
  socket, or a vsock fd into a Nether guest, registers into the reactor the same
  way PG sockets and proxy upstreams do.
- **Build posture:** `linkSystemLibrary` for OpenSSL/zlib, native-gated; no
  vendored C source yet. wasm3 would be the first vendored C dependency; zware
  would avoid C entirely. This is a real consideration for the runtime choice.

## Design sketch

### Module lifecycle

Load and validate the `.wasm` at config time (startup or SIGHUP reload, never the
hot path). Parse/compile once per worker. Pre-instantiate a pool of N instances
per module per worker so the hot path only *acquires* an instance, never builds
one.

### Instance pool

`[N]WasmInstance`, each owning a fixed linear memory (cap configurable, e.g.
1-4 MB), a free-stack for acquire/release, and a state: `idle`, `running`,
`parked` (host call in flight, withheld from reuse, mirrors a busy PG slot).
Per-worker, not per-connection: total memory is N * mem_cap per module per
worker, fixed and known at startup.

### ABI (custom, minimal, v1)

Host imports the filter may call:

- `get_method() -> i32` (method enum), `get_path(ptr, cap) -> len`,
  `get_header(name_ptr, name_len, out_ptr, out_cap) -> len`, `header_count()`.
  Headers are copied host -> linear memory *on demand, one at a time*: a filter
  that reads two headers copies two, not the whole block. This is how the
  zero-copy tax is kept proportional to what the filter actually touches.
- `set_response_header(name_ptr, name_len, val_ptr, val_len)` and
  `respond(status, body_ptr, body_len)` for the reject path (staged, applied when
  `on_request` returns the reject code).
- `host_call(...)` async (Phase 3): stage an outbound request, return a token,
  yield. Result delivered on resume.
- `log(ptr, len)`.

Filter export: `on_request() -> i32` mapping to `Decision`:
`0 = allow`, `1 = reject` (uses staged status/body), `2 = modify` (uses staged
headers), plus a reserved `3 = parked` for Phase 3 host calls.

### Async host calls and the Nether bridge (Phase 3)

`host_call` stages the request and returns a "want resume" signal; `on_request`
returns the `parked` code; the instance enters `parked` (pinned, not returned to
the pool); swerver parks the HTTP request (reuse the `.db_parked`-style
connection state, or add `.wasm_parked`); the outbound call (or the vsock send
into a Nether guest) completes on the reactor; resume re-enters the instance at a
resume export with the result copied into linear memory. This is the design 9.0
park-and-resume contract with the **pinned instance standing in for the stash**
(a WASM linear memory does not fit in 256 bytes). The host-call target may be an
HTTP upstream or a **vsock address into a Nether microVM**: that is the two-tier
model made concrete, a Tier-1 filter mediating and metering a Tier-2 guest over
the spine Nether already speaks.

### Resource bounding (security-critical)

An op-count/fuel budget checked in the interpreter loop, interrupting at the
budget. A linear-memory growth cap. A per-invocation wall-clock deadline enforced
from the housekeeping tick, exactly like `statement_timeout_ms` for PG. Because
the reactor is single-threaded, a runaway filter blocks every connection on the
worker, so this is mandatory and gets its own test surface. wasmtime provides
fuel/epoch interruption natively; wasm3 needs an op-count hook; zware varies.
This requirement weighs on the runtime choice.

### Config

A route binds a WASM module path plus limits (mem cap, fuel budget, timeout),
following the shape of the `postgres` and `x402` config blocks. Per-route or
global, like the middleware chain.

## Test plan

- **Phase 0 benchmark:** ns/op for a trivial filter vs native middleware. This
  number decides the whole effort and the runtime choice.
- **Conformance:** golden filters (header-add, reject-on-missing-auth,
  route-rewrite) with golden request/response pairs.
- **Resource bounding (the security tests):** a runaway-loop module is
  interrupted at the fuel budget without wedging the worker; a memory-bomb module
  hits the cap and traps rather than OOMing the worker; a wall-clock timeout
  fires from the tick. If any of these fails, the feature is unshippable.
- **Park lifecycle:** host_call completes / times out / the client disconnects
  while parked (generation check, instance released on cancel). Mirror the PG
  park tests.
- **ABI fuzz:** a filter passing garbage ptr/len to `get_header`/`respond` must
  trap, never read or write outside its linear memory. This boundary is the
  guest-to-host trust surface; fuzz it the moment it exists.
- **Nether bridge (once Nether exists):** filter -> vsock -> guest round trip
  through park-and-resume.

## Open questions

1. **Runtime.** ~~wasm3 vs zware vs wasmtime.~~ **RESOLVED (Phase 0): wasm3.**
   The spike showed wasm3 keeps the single static binary (contrary to the
   original worry), is ~3x faster than zware, has a proven near-free fuel patch,
   and is immune to Zig-version churn. See `spike/wasm/RESULTS.md`.
2. **Custom ABI vs proxy-wasm.** v1 is custom and minimal. proxy-wasm is gated on
   the Phase 0 perf result *and* real ecosystem demand. Is running unmodified
   Envoy filters ever an actual customer ask, or is it vanity compatibility?
3. **Park state.** Reuse the connection park enum with a `.wasm_parked` value
   (one byte, free, like `.db_parked`) vs a dedicated mechanism. Lean reuse.
4. **Instance across park.** Pin the instance (simple, caps concurrent parked
   filters at the pool size, matches the busy-slot model) vs snapshot linear
   memory into a park buffer (complex, frees the instance). Lean pin for v1.
5. **Demand gate.** What concrete external signal greenlights Phase 1? Working
   definition: a customer wants to run their own code on the hot path. Until then,
   Phase 0 spike only.
6. **Pool topology.** Per-module vs per-route instance pools; N sizing per
   worker; behavior on cold module load during SIGHUP reload.
7. **Is the cheap tier ever the *workload*, not just the mediator?** If an edge
   function is sometimes light agent compute (not only a gateway filter), the ABI
   needs more (a WASI-lite surface, KV, fetch) and this becomes a Tier-1 *product*
   rather than a filter mechanism. If it stays filter-shaped, keep the ABI tiny.
   This ties directly to how the platform defines its Tier-1 offering against
   Nether's Tier-2, and is the most strategic of these questions.
