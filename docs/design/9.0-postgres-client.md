# Native PostgreSQL Client

Status: implemented (alpha.23). Last verified: 2026-06-12 (alpha.23).
Owner: data subsystem. Phases 1–2 done (per-worker pool, SCRAM-SHA-256,
TLS client mode, io_uring external-FD dispatch, park-and-resume handler
API); pipelining infra present (phase 4 — op FIFO + prepared-statement
cache); TechEmpower perf tuning ongoing (phase 5). The design rationale
below is preserved as written during scoping (2026-06-10); status callouts
have been corrected.

## Context

swerver can route, authenticate, rate-limit, cache, and proxy — but every
dynamic response requires an upstream HTTP service. A native PostgreSQL
client closes the gap between "API gateway" and "app server": handlers
that serve data directly, with no sidecar process. Secondary motivations:

1. **TechEmpower.** The TFB database rounds (single-query, multi-query,
   fortunes, updates) require a PG client, and the leaderboard is won on
   *pipelined* extended-protocol usage. swerver's shape — single-threaded
   reactor per core, multi-process fork — is exactly the architecture
   that tops those charts when paired with a per-worker pipelined PG
   connection.
2. **Internal consumers.** x402 settlement persistence (today an
   append-only JSONL fallback), durable rate-limit state, admin API
   config storage all fall out for free.
3. **Ethos fit.** We wrote our own QUIC stack, HPACK, and QPACK; the PG
   wire protocol v3 is far simpler than any of them. Binding libpq would
   add a C dependency and surrender control of the I/O model, which is
   the whole point.

The binding constraint is the reactor. The proxy's blocking upstream I/O
is already identified as the deeper problem in design 5.0; a PG client
must be **non-blocking and event-driven from day one** — a slow or
unreachable database must never stall non-DB routes on the same worker.
That forces the one genuinely new design surface: swerver has no async
handler API. The x402 settle-park mechanism (response held on the
connection, reactor polls a completion, response flushed on arrival) is
the in-house precedent and this design generalizes it.

TLS is **in scope for v1** (decision 2026-06-10): managed Postgres (RDS,
Neon, Supabase, Cloud SQL) requires TLS, and a client that cannot reach
them undercuts the feature's reason to exist. Certificate verification is
mandatory — the facilitator client's missing cert verification is an open
HIGH audit finding; the PG client does not repeat it.

## Goals

- PostgreSQL wire protocol v3 client in Zig: startup, auth, extended
  query protocol (Parse/Bind/Execute/Sync), prepared statements,
  ErrorResponse/NoticeResponse handling.
- Fully event-loop integrated: PG sockets are reactor citizens on all
  four backends (kqueue, epoll, io_uring_poll, io_uring_native). Zero
  blocking calls on the reactor — including connect and TLS handshake.
- **TLS in v1** with `verify-full` semantics (chain + hostname), client
  mode added to the existing OpenSSL provider. `sslmode` config:
  `disable | require | verify-full`; default `verify-full` when TLS is on.
- SCRAM-SHA-256 auth (the modern default; OpenSSL EVP primitives are
  already linked). Cleartext password permitted only over TLS.
- Pipelining: multiple queries in flight per connection, FIFO
  demultiplexing (extended-protocol responses are strictly ordered).
- Binary result format for the common OIDs (bool, int2/4/8, float4/8,
  text, varchar, bytea, timestamptz, uuid, numeric-as-text); text format
  fallback for everything else.
- Per-worker connection pool (small, e.g. 2–4 conns), per-worker prepared
  statement cache. No cross-worker sharing — workers are separate
  processes.
- An async handler API (park-and-resume) that x402 settle-park can later
  migrate onto.
- Zero per-query heap allocation on the fast path: requests serialize
  into pool buffers, rows parse into the per-request arena.

## Non-goals (this effort)

- COPY in/out, LISTEN/NOTIFY, cursors/portals beyond the unnamed portal,
  savepoints, multi-host failover / load-balanced DSNs.
- Query builder, ORM, migrations — raw parameterized SQL only.
- A blocking convenience API. There is deliberately no `querySync()`.
- Channel binding (SCRAM-SHA-256-PLUS) — accept plain SCRAM over TLS in
  v1, revisit.
- md5 auth (legacy; off by default, decide later if demand appears).
- Replacing the x402 JSONL settle log with PG (consumer, not part of this
  effort).

## Alternatives considered (2026-06-10)

**Tokio/Rust sidecar** (tokio-postgres + deadpool behind a unix socket,
or proxying to PostgREST). Rejected. It saves only the low-risk half of
the work (the wire codec, SCRAM, TLS — well-documented and unit-testable
against dockerized PG) while keeping the high-risk half: swerver still
needs non-blocking upstream I/O, request parking, and continuation
dispatch regardless of what's on the other end of the socket. Meanwhile
it adds an IPC hop on every query (fatal for the TFB story; no top entry
does DB access out-of-process), a second protocol to parse, a
work-stealing multithreaded runtime competing with pinned per-core
workers, and a second supervised artifact against the single-static-
binary ethos. Note: proxying to PostgREST via the *existing* reverse
proxy is a legitimate zero-code experiment to validate demand for
DB-backed routes before phase 2 starts — but it is a demo, not the
feature.

**Async libpq** (the strongest alternative — `PQsendQueryParams` /
`PQconsumeInput` is non-blocking, exposes its socket FD for reactor
integration, pipeline mode since PG14, OpenSSL already a dependency).
Rejected on narrower grounds: PGresult heap-allocates per result, which
clashes with the zero-alloc fast path; it adds a C dependency for
exactly the layer (a wire codec) this codebase is best equipped to own;
and it would not have dodged the actual risk anyway — its async mode
hands back the same readiness-driven resumption model as a native
client, with C ownership semantics on top.

**Where the risk actually concentrates.** Zig has no language-level
async, but swerver has never used or needed it — H2 stream states, the
QUIC handshake/loss machinery, TLS-over-memory-BIO, and x402 settle-park
are all explicit reactor-driven state machines, and the PG connection
state machine is a strictly simpler member of that genre than QUIC. The
genuinely new risk is that park-and-resume is the first state machine
handed to *handler authors* rather than kept behind the dispatch loop:
the failure mode is third parties violating park invariants
(borrow-after-park, resume-after-timeout, missing stash), not us failing
to build the machinery. Mitigation: the framework enforces the
invariants mechanically — typed park tokens, the generation check built
into resume rather than left to handlers, debug-mode assertions on
borrowed slices — and open question 1 (the blessed continuation pattern)
is treated as the most important design decision in this document.
Explicit state machines also stay forward-compatible with any future
Zig async story (`std.Io` direction): sugar can be layered over the same
parking machinery later.

## Architectural survey (what exists to build on)

- **Parking precedent** — x402 settle-park: response bytes held in
  `conn.x402_held_buf` + metadata scalars, facilitator completion polled
  from the reactor, response flushed on arrival (`router.zig:873`,
  `x402_client.zig`). Generalizes to "handler suspends awaiting an
  external completion."
- **External-FD tagging** — design 5.0 proposes `UPSTREAM_ID_BIT |
  client_conn_index` for proxy upstream sockets; the 64-bit conn-id space
  is opaque to all backends (`io.zig registerConnection(conn_id, fd)`).
  PG sockets claim a second tag bit. Whichever effort lands first builds
  the shared "non-Connection FD" dispatch plumbing.
- **Buffer pool** — slab-backed `BufferHandle` acquire/release, 16KB
  buffers + 1MB body buffers (`buffer_pool.zig`). PG send/recv buffers
  come from here.
- **Per-request arena** — 16KB `FixedBufferAllocator`, lazily acquired,
  freed after response serialization (`router.zig:82`). Decoded rows for
  a parked request must NOT live here (it dies with the handler frame) —
  see Design sketch.
- **TLS provider** — OpenSSL via FFI, server-mode only today
  (`provider.zig initTcp/initQuic/initTcpSniMtls`). Client mode is new
  but the FFI surface (SSL_CTX, BIO, verify callbacks) is already bound.
- **What to avoid** — proxy's `connectBlocking`/`sendAll`/`recvBlocking`
  on the reactor (design 5.0's motivating wart) and the facilitator's
  unverified TLS (audit HIGH).

## Design sketch

### PgConn state machine

The implemented `SlotState` (`src/db/pg/client.zig`) has eight states:

```
closed → connecting → ssl_request → tls_handshaking → handshaking → ready ⇄ busy → closed
                 (skip ssl_request/tls_handshaking if sslmode=disable)
[any] → failed → backoff → connecting
```

- `connecting`: non-blocking `connect()`, FD registered for writability.
- `ssl_request`: `SSLRequest` (8 bytes) sent; awaiting the server's
  single-byte `'S'`/`'N'` verdict before starting TLS.
- `tls_handshaking`: TLS ClientHello onward over the memory-BIO session,
  driven by readiness events through the same machinery the server side
  uses. (Both SSL states are skipped under `sslmode=disable`.)
- `handshaking`: startup message, SCRAM exchange
  (client-first/server-first/client-final/server-final), then
  `ReadyForQuery`.
- `ready/busy`: pipelined operation, below.
- `failed`: a distinct error path (TLS refusal, auth failure, connection
  loss); the slot backs off and re-enters `connecting`.

### Event-loop integration

PG sockets register with id `PG_ID_BIT | (worker_pg_slot)`. The dispatch
loop routes those events to the PG client module instead of the
connection table. All four backends already treat conn ids as opaque, so
per-backend work is the same dispatch-switch addition design 5.0 phase 1
needs — shared infrastructure, built once.

### Pipelining and demux

Built (phase 4): per-worker pool with park/resume, a pipelining op FIFO
(`OP_QUEUE_DEPTH = 16` per connection), and a per-connection
prepared-statement cache (`STMT_CACHE_SIZE = 32`, named statements
`s00..s31`). The original sketch:

Per connection: a FIFO of in-flight `PgOp`s (fixed ring, e.g. 32 deep).
Each op = Parse(named or cached)/Bind/Execute/Sync serialized into the
send buffer; responses (ParseComplete…CommandComplete/ErrorResponse…
ReadyForQuery) are consumed in order and matched to the FIFO head.
ErrorResponse aborts ops queued behind it in the same implicit
transaction until the matching ReadyForQuery; those complete with
`error.PipelineAborted`. Prepared statements are cached per connection
keyed by SQL hash; cache hit skips Parse.

### Handler API (decided 2026-06-10)

Two-phase handler, generalizing x402 settle-park. The design goal is
that every park invariant is enforced by the type system or the
framework, not by documentation — this is the surface we hand to
third-party handler authors (see Alternatives considered: that exposure
is where the project risk concentrates).

**1. The park sentinel IS `query()`'s return value.** `query()` returns
the parked `Response` itself (a reserved marker the router intercepts
before serialization). One expression issues the query, registers the
continuation, and produces the only value that parks the connection —
"issued a query but forgot to park" and "parked with nothing in flight"
are unwritable:

```zig
fn listUsers(ctx: *HandlerContext) Response {
    return ctx.pg.query(
        "select id, name from users where org_id = $1",
        .{ctx.param("org")},
        Stash, .{ .org_id = org_id },
        onUsers,
    ) catch |err| switch (err) {
        error.QueueFull, error.NotConnected => Response.serviceUnavailable(),
    };
}
```

Synchronous failures (pool down, per-connection FIFO full) surface as
ordinary errors the handler maps to a 503 *now*, while it still has the
request. The signature stays flat (sql, args, stash type, stash init,
continuation) — no builder indirection on a hot path (decided; the
five-argument arity is the cost and it is acceptable).

**2. Continuations receive a distinct `ResumeContext` — with no
`request` field.** The read buffer is recycled while parked; reusing
`HandlerContext` with dead request slices would be the borrow-after-park
trap waiting to happen. A separate type makes "phase 2 reads request
headers" a compile error:

```zig
pub const PgContinuation = *const fn (rctx: *ResumeContext) Response;

pub const ResumeContext = struct {
    result: PgError!Result,   // Timeout, ConnectionLost, PipelineAborted,
                              // ServerError, ResultTooLarge
    response_buf: []u8,       // fresh scratch, as in HandlerContext
    response_headers: []response.Header,
    // arena + app_state/services accessors — same as HandlerContext
    // pg access for chaining (see 4)
    // NO request field.
};
```

The continuation runs **exactly once** — success, server error, and
timeout all deliver through `result` — unless the HTTP client
disconnected while parked, in which case the generation check (built
into resume, same pattern as `tunnel_peer_id`) drops the completion and
frees the op. Error handling is explicit per continuation:
`rctx.result catch` is one line, and a route-level error default was
considered and rejected — DB failure handling belongs where the
response is built (decided).

**3. The stash is typed and comptime-rejected if it contains
pointers.** Phase 1 must copy out anything phase 2 needs; the framework
makes that mechanical. `query()` runs `comptime assertPlainData(Stash)`
— any pointer or slice field is a compile error explaining why. Copying
request bytes into fixed arrays is forced by the type system:

```zig
const Stash = struct {
    org_id: u64,
    name_buf: [64]u8,   // copied bytes — a slice would not compile
    name_len: u8,
    step: u8 = 0,
};
```

Capped at 256 bytes; lives in the connection's park state alongside the
x402-style held metadata. `rctx.stash(Stash)` returns the typed pointer.

**4. Multi-query = one continuation + a `step` switch (the blessed
pattern).** A continuation may itself return a parked `Response`, so
chains re-enter the same machinery. The canonical shape is a single
named function switching on a stash `step` field — no framework helper
(decided; resolves open question 1). It reads like a state machine
because it is one:

```zig
fn onUserFlow(rctx: *ResumeContext) Response {
    const st = rctx.stash(Stash);
    switch (st.step) {
        0 => {
            const user_id = (rctx.result catch return dbError(rctx)).row(0).int8(0);
            st.user_id = user_id;
            st.step = 1;
            return rctx.pg.query("select * from orders where user_id = $1",
                                 .{user_id}, onUserFlow) catch Response.serviceUnavailable();
        },
        1 => return renderOrders(rctx),
        else => unreachable,
    }
}
```

**5. Results are borrow-only, with debug-mode poisoning.**
`Result.rows()` iterates `DataRow` slices borrowing the PG connection's
recv buffer — zero-copy (resolves open question 3). Valid only inside
the continuation call; serialize into `response_buf`/arena before
returning. Debug builds memset the result region to `0xAA` after the
continuation returns, so an escaped slice fails loudly in tests instead
of silently in production.

**Scoping:** H1-first, same as x402 async verify — a parked H1 request
pauses reads on its connection (H1 responses are ordered anyway). H2
per-stream parking is a follow-up with its own stream-state
interaction. x402 settle-park migrating onto this machinery is likewise
a follow-up, not v1.

### TLS client mode

`provider.zig` gains `initTcpClient(host, ca_bundle, sslmode)`:
`SSL_VERIFY_PEER` + `SSL_set1_host` for hostname verification, system CA
bundle by default with a config override (`ssl_root_cert`). `require`
skips verification (explicitly discouraged in docs), `verify-full` is the
default. Handshake is driven through memory BIOs by reactor readiness,
same as server-side sessions.

### Memory model

- Send path: one pool buffer per connection, ops serialize in place,
  flushed by writability events. No heap.
- Recv path: one pool buffer per connection; a `DataRow` spanning the
  buffer boundary grows into a body buffer (1MB) — same overflow pattern
  as proxy responses. Result sets larger than that complete with
  `error.ResultTooLarge` (configurable cap); v1 does not stream rows to
  handlers.
- Row decode: lazy. `PgResult.row(i).col(j)` returns slices into the recv
  buffer; binary decode helpers (`asInt64`, `asText`, …) on access.

### Config

```json
"postgres": {
  "url": "postgres://user@host:5432/db?sslmode=verify-full",
  "password_env": "PG_PASSWORD",
  "pool_size_per_worker": 2,
  "statement_timeout_ms": 5000,
  "ssl_root_cert": "/etc/ssl/rds-ca.pem"
}
```

Password via env var only — never in the config file (it is served by the
admin API). `pool_size_per_worker` must be 1–4 (config validation rejects
0 or >4 with `error.InvalidPostgresConfig`); default 2. `statement_timeout_ms`
defaults to 5000 and bounds the parked-op deadline.

## Test plan (acceptance)

- **Protocol**: codec round-trip unit tests; auth matrix against
  dockerized PG 14/15/16/17 (SCRAM, cleartext-over-TLS); error paths
  (bad password, missing database, syntax error).
- **TLS**: verify-full positive (valid chain + hostname); negative —
  self-signed cert, hostname mismatch, server without TLS while
  `sslmode=require` — all must fail closed with distinct errors.
- **Reactor isolation** (the headline invariant): with PG paused
  (`docker pause`), non-DB routes on the same worker must show no p99
  shift; DB routes time out cleanly at the configured deadline.
- **Pipelining**: N queries in flight, kill the PG connection mid-stream
  — every parked HTTP request gets a 5xx, no leaked connections, pool
  reconnects; ErrorResponse mid-pipeline aborts trailing ops with
  `PipelineAborted` and recovers at ReadyForQuery.
- **Park lifecycle**: client disconnects while parked (continuation must
  not write to a recycled connection — generation-id check, same pattern
  as tunnel_peer_id); park timeout fires response; stash survives read
  buffer recycling.
- **Perf**: TFB-style single-query / multi-query(20) / fortunes / updates
  profiles vs a libpq-based reference (e.g. nginx+app or fastify+pg) on
  the Linux box; target is parity-or-better at equal core count.

## Open questions

1. ~~**Continuation ergonomics**~~ — RESOLVED 2026-06-10: no framework
   helper; the single-continuation + stash `step` switch is the blessed
   pattern, documented in the Handler API section above.
2. ~~**Shared external-FD plumbing**~~ — RESOLVED 2026-06-10: PG built
   it (`EXTERNAL_ID_BIT`, bit 62, exact-match `isExternalId` — see
   `src/runtime/io.zig`); landed for kqueue/epoll in phase 2.1
   (`b86f259`/`b4c551c` on pg-client). Proxy streaming (design 5.0)
   adopts it. io_uring backends are the explicit next step.
3. ~~**Result ownership**~~ — RESOLVED 2026-06-10: borrow-only rows,
   serialization inside the continuation, debug-mode poisoning of the
   result region after the continuation returns. Consumers needing rows
   beyond the call copy fields into their stash or arena explicitly.
4. **io_uring_native** — poll-driven PG sockets in v1, or full SQE-based
   send/recv from the start? (Poll-driven is simpler and uniform across
   backends; native SQE is a perf follow-up.) Phase 2.1 survey notes:
   the native backend's u4 op space has room for an `external_poll`
   kind; the poll backend needs a small external-fd re-arm table.
5. **Pool sizing** — fixed per-worker pool vs queue-depth-adaptive. v1
   says fixed; revisit with TFB data.
6. **numeric** — text-format passthrough in v1 (decode to string), or
   binary numeric decode (non-trivial)? Leaning text.
