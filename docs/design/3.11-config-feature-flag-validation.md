# Core Runtime: Config and Feature-Flag Validation

Reference core runtime: [3.0-core-runtime.md](3.0-core-runtime.md).
Reference contracts: [1.4-contracts.md](1.4-contracts.md).

## Scope

Defines validation rules for configuration and build-time feature flags to ensure safe runtime behavior.

## Build-Time Flags

- `enable-tls`
- `enable-http2`
- `enable-http3`
- `enable-proxy`
- `enable-io-uring`

Rules:

- `enable-http3` requires `enable-tls` and a QUIC provider.
- `enable-io-uring` is Linux-only; ignored on other platforms.
- Feature flags should gate code paths at compile time to avoid unused dependencies.

## Runtime Config Validation

Rules:

- `max_connections > 0`.
- `buffer_pool.buffer_count >= max_connections * 2` to guarantee 1 read + 1 write buffer per connection.
- `backpressure.read_high_water` and `backpressure.write_high_water` must be multiples of `buffer_size`.
- `backpressure.read_low_water` and `backpressure.write_low_water` must be <= their high-water values.
- `timeouts` must be non-zero; `header_ms <= idle_ms`.
- `postgres.pool_size_per_worker` must be `1..4` when PostgreSQL is enabled. A value of `0` or `> 4` is **rejected** with `error.InvalidPostgresConfig` (it is not clamped). PostgreSQL config also requires non-empty `host` and `user`.

## Parsed / Validated Config Keys

In addition to the limits above, the config parser recognizes the following newer keys:

- `server.cache_static_files` (bool): enable the in-memory static file cache.
- `server.preencoded` (bool, positive key, default on): keep the preencoded response cache; maps internally to `disable_preencoded`.
- `server.listeners[]`: explicit per-port listeners. Each entry is `{ address, port, use_tls, h2c_only, quic_enabled, quic_port }`. When present, the worker binds each via `SO_REUSEPORT`; otherwise a single listener is synthesized from the legacy single-port fields.
- `postgres.{ url, password_env, pool_size_per_worker (1-4), statement_timeout_ms }`: the password is read from the environment variable named by `password_env`, never from the config file (a password in the `url` is ignored).

## Startup Behavior

- Validate config on startup and fail fast with descriptive errors.
- Emit selected feature flags and limits in startup logs.

## Validation

- Unit tests for config validation.
- Build-time checks ensure invalid flag combinations are rejected.
